Docker

  1. docker pull：用于从 Docker 镜像仓库下载镜像到本地主机。

    • 语法：docker pull [OPTIONS] NAME[:TAG|@DIGEST]
    • 示例：docker pull ubuntu:20.04

    这个命令用于获取 Docker 镜像，其中 NAME 表示镜像的名称，TAG 表示镜像的标签，或者可以使用 @DIGEST 来指定镜像的摘要。如果不指定标签，将默认获取 latest 标签的镜像。

  2. docker load：用于从文件加载镜像，通常是以 tar 归档文件的形式。

    • 语法：docker load [OPTIONS]
    • 示例：docker load -i myimage.tar

    这个命令用于加载以 tar 归档文件格式存储的 Docker 镜像。通常用于将镜像从一个环境传输到另一个环境。

  3. docker save：用于将 Docker 镜像保存为 tar 归档文件。

    • 语法：docker save [OPTIONS] IMAGE [IMAGE...]
    • 示例：docker save -o myimage.tar myimage:tag

    这个命令允许你将一个或多个 Docker 镜像保存为 tar 归档文件，通常用于导出镜像以便在不同环境中使用。

  4. docker run：用于创建和启动 Docker 容器。

    • 语法：docker run [OPTIONS] IMAGE [COMMAND] [ARG...]
    • 示例：docker run -d -p 8080:80 my-web-app

    这个命令用于在 Docker 容器中运行一个镜像，你需要指定镜像的名称、容器选项（如端口映射、环境变量、容器名称等），以及容器内要执行的命令。

  5. docker build：用于构建自定义 Docker 镜像，通常通过 Dockerfile 文件定义构建步骤。

    • 语法：docker build [OPTIONS] PATH | URL | -
    • 示例：docker build -t my-custom-image .

    docker build 命令用于基于 Dockerfile 构建自定义镜像。你需要提供包含 Dockerfile 的路径，以及可选的选项和标签。

    Dockerfile

  6. docker rm：用于删除已停止的容器。

    • 语法：docker rm [OPTIONS] CONTAINER [CONTAINER...]
    • 示例：docker rm my-container

    这个命令用于删除一个或多个已经停止的容器，释放资源并清理系统。

  7. docker stop：用于停止运行中的容器。

    • 语法：docker stop [OPTIONS] CONTAINER [CONTAINER...]
    • 示例：docker stop my-container

    这个命令用于停止一个或多个正在运行的容器，使它们从运行状态转换为停止状态。

  8. docker ps：用于列出正在运行的容器。

    • 语法：docker ps [OPTIONS]
    • 示例：docker ps -a

    docker ps 命令用于列出当前正在运行的容器，包括容器的 ID、状态、端口映射等信息。加上 -a 选项可以列出所有容器，包括已停止的容器。

Git操作实例

Git 的设计是基于不可变的 DAG（有向无环图）

工作区：开发者在工作区中进行日常的代码开发和修改工作（就是什么git操作都没有做直接改代码，这个地方就是工作区

暂存区：git add 会将文件的当前状态（即 修改后的内容）从 工作区 拷贝到 暂存区。如果此时再次修改 工作区 内容， 暂存区 内容不受影响。

https://learngitbranching.js.org/

branch

创建一个新的分支指针，并将它指向当前的 HEAD 所在的检查点（即当前的 commit）。这个新的分支指针会指向当前分支的最新提交，或者如果你在一个 detached HEAD 状态下，它会指向 HEAD 当前指向的 commit。

  • 分支指针始终指向该分支最新的 commit（HEAD 的位置）。

  • 当在该分支上提交新的 commit 时，分支指针会自动前移到新 commit。

  • 注意，分支指针指向的是最新的 commit，而不是所有 commit。历史 commit 通过链式引用联系在一起。

commit

创建检查点，每个 commit 都有一个或多个后继引用它（例如分支或标签），但它不“属于”任何特定分支。

判断一个节点是否是“野生”的，核心就是看是否有任何分支指针通过回溯能够遍历到它。

add

将工作区内容覆写到暂存区

checkout

使head指针指向指定branch指针或检查点

git checkout main # 切换head指向main分支指针

git checkout C1 # 切换head指向检查点，此时head指针与main指针分离！！！，后续提交的节点为"野生"节点

当 HEAD 指针移动到 main 分支上的非最新提交 c1 (最新提交是 c2) 后，执行commit指令

  1. HEAD 指针移动: git checkout c1 命令会将 HEAD 指针移动到 c1 提交，此时处于 “分离 HEAD 状态”。
  2. 新提交产生: 当进行新的提交时，Git 会创建一个新的提交对象 c3，并将 c1 作为 c3 的父提交。
  3. HEAD 指针移动: c3 提交创建后，HEAD 指针会指向新的提交 c3。新的检查点将不属于任何branch，可能在未来的被垃圾回收

reset

移动当前branch指针到指定检查点

假设有以下历史：

  1. git reset --soft B：
    • HEAD 指向 B。
    • C 和 D 被“取消”，但它们的更改仍然在暂存区中（即提交C和D的修改内容全部保存在暂存区
  2. git reset --mixed B：
    • HEAD 指向 B。
    • C 和 D 的更改从暂存区移到工作区。
  3. git reset --hard B：
    • HEAD 指向 B。
    • C 和 D 被丢弃，暂存区和工作区都回到 B 的状态。

pull

  • pull指令为fetch和merge指令的和，因此会使当前branch指针移动
  • git pull 拉取时，默认是针对当前分支关联的单个远程分支进行操作

git pull 的一些常见行为：

  • git pull (不带参数): 这是最常用的方式。它会从当前分支追踪的远程分支获取最新提交，并尝试将其合并到当前分支。如果你的当前分支已经设置了上游分支 (upstream branch)，那么 git pull 将会使用这个上游分支。你可以使用 git branch -vv 查看当前分支追踪的远程分支。

  • **git pull <remote> <branch>: ** 这会从指定的远程仓库 <remote> 的 <branch> 分支获取最新提交，并尝试将其合并到当前分支。 例如，git pull origin develop 会从远程仓库 origin 的 develop 分支获取更新，并合并到当前分支。

  • git pull --rebase: 这会使用 git rebase 而不是 git merge 来整合远程分支的更改。rebase 会将你的本地提交放在远程分支的最新提交之后，从而创建一个更线性的提交历史。 可以通过 git config --global pull.rebase true 设置全局使用 rebase 模式。

push

  • git push 命令默认只推送当前分支到远程对应的分支。

几种 git push 的行为：

  • git push (不带任何参数): 推送当前分支到其upstream分支。如果当前分支没有设置upstream分支，会报错。 这是最常用的方式。

  • git push <remote>: 推送当前分支到指定远程仓库 <remote> 的同名分支。例如 git push origin 会将当前分支推送到名为 origin 的远程仓库的同名分支。

  • git push <remote> <branch>: 将本地 <branch> 分支推送到远程仓库 <remote> 的 <branch> 分支。例如 git push origin develop 会将本地的 develop 分支推送到远程仓库 origin 的 develop 分支。

  • git push <remote> <local_branch>:<remote_branch>: 将本地 <local_branch> 分支推送到远程仓库 <remote> 的 <remote_branch> 分支。 这允许你将本地分支推送到远程的不同名分支。例如 git push origin feature/new-login:develop 会将本地的 feature/new-login 分支推送到远程仓库 origin 的 develop 分支。

  • git push <remote> --all: 推送所有本地分支到远程仓库 <remote>。

  • git push <remote> --tags: 推送所有本地标签到远程仓库 <remote>。

Makefile

使用Make指令执行Makefile文件(make指令自动寻找当前文件夹下的Makefile文件)

Makefile文件

冒号左侧：目标文件

冒号右侧：被依赖文件，前置条件

前置条件决定 “目标” 是否需要重新构建。只要前置条件中列出的文件中，有任何一个文件被更新（即前置条件列出的文件的最后修改时间比目标的最后修改时间新），或者 “目标” 不存在，那 “目标” 就需要重新构建。

如果前置条件中列出的文件中有任何一个文件不存在，那么 Make 就需要在 Makefile 文件中寻找并执行能够生成该文件的规则，也就是说缺失文件本身也是一个 “目标”，需要先将该 “目标” 构建出来才能利用它去构建当前 “目标”（构建 A 时发现 A 需要依赖 B，所以要先构建 B 才能继续构建 A，简称递归生成依赖）。

$@，$^，$< 这种符号，这种符号称为自动变量。自动变量是局部变量，作用域范围在当前的规则内（即自动化变量只应该出现在 Makefile 目标规则中）

$@ 指代当前目标， Make 命令意图构建的目标

$< 指代第一个前置条件。比如规则为 hello: a b，那么 $< 就指代 a。

$^ 指代所有前置条件，之间以空格分隔。比如，规则为 hello: a b，那么 $^ 就指代 a 和 b。
