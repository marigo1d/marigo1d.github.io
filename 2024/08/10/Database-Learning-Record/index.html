SQL LIKE 运算符用于模式匹配，允许你搜索包含特定字符序列的数据。LIKE 运算符通常与通配符一起使用，以创建更灵活的搜索模式。以下是常用的 LIKE 通配符：

  • % (百分号): 匹配任意字符序列（包括空序列）。它可以用于匹配字符串的开头、结尾或中间的任意部分。

    • 示例:
      • LIKE 'A%': 匹配以 “A” 开头的任何字符串，例如 “Apple”、”Airplane”、”A”。
      • LIKE '%e': 匹配以 “e” 结尾的任何字符串，例如 “Apple”、”Orange”、”e”。
      • LIKE '%app%': 匹配包含 “app” 的任何字符串，例如 “Apple”、”Application”、”Happy”。

  • _ (下划线): 匹配任何单个字符。

    • 示例:
      • LIKE 'A_': 匹配以 “A” 开头且后面跟着一个任意字符的字符串，例如 “Ab”、”Ac”、”A1”。
      • LIKE '_pple': 匹配以任意字符开头且后面跟着 “pple” 的字符串，例如 “Apple”、”ipple”。
      • LIKE 'a__le': 匹配以 “a” 开头，后面跟着两个任意字符，再跟着 “le” 的字符串，例如 “apple”、”azble”。

  • [] (方括号): 匹配指定集合中的任何单个字符。

    • 示例:
      • LIKE '[ABC]%': 匹配以 “A”、”B” 或 “C” 开头的任何字符串。
      • LIKE 'h[aeiou]t': 匹配 “hat”、”het”、”hit”、”hot” 或 “hut”。
      • LIKE '[A-Z]%': 匹配以任何大写字母开头的字符串。

  • [^] (方括号加脱字符): 匹配不在指定集合中的任何单个字符。

    • 示例:
      • LIKE '[^ABC]%': 匹配不以 “A”、”B” 或 “C” 开头的任何字符串。
      • LIKE 'h[^aeiou]t': 匹配以 “h” 开头，第二个字符不是元音字母，以 “t” 结尾的字符串。

数据库事务的 ACID 特性

ACID 是数据库事务的四个重要特性，它们保证了事务的可靠性和一致性。

1. 原子性 (Atomicity)

  • 定义: 一个事务的所有操作要么全部成功执行，要么全部不执行，不会出现部分成功部分失败的情况。
  • 作用: 保证数据库状态的一致性，即使事务执行过程中发生错误，数据库也能回滚到事务开始前的状态，就像什么都没发生过一样。
  • 举例: 转账操作需要从账户 A 扣款并向账户 B 加款，这两个操作必须作为一个整体执行，要么都成功，要么都失败。如果只扣款成功而加款失败，就会导致数据不一致。

2. 一致性 (Consistency)

  • 定义: 事务执行前后，数据库必须保持一致的状态，满足所有预定义的约束和规则。
  • 作用: 保证数据库数据的完整性和正确性，防止数据出现逻辑上的错误。
  • 举例: 银行账户的余额不能小于零。如果一个事务导致账户余额变为负数，则该事务违反了一致性约束，数据库会回滚事务，保持数据一致性。

3. 隔离性 (Isolation)

  • 定义: 多个事务并发执行时，彼此之间相互隔离，互不干扰，就像每个事务都是独立执行的一样。
  • 作用: 防止并发事务之间的数据冲突，保证每个事务都能看到一致的数据。
  • 举例: 用户 A 和用户 B 同时读取账户 C 的余额，然后分别进行转账操作。隔离性保证了用户 A 和用户 B 的操作不会相互影响，每个用户都能看到正确的账户余额。

4. 持久性 (Durability)

  • 定义: 一旦事务提交成功，其对数据库的修改将永久保存，即使数据库发生故障也不会丢失。
  • 作用: 保证数据的可靠性和安全性，即使系统崩溃，数据也能恢复到最新状态。
  • 举例: 用户 A 成功转账后，即使数据库突然断电，转账操作的结果也不会丢失，数据库重启后，账户 A 和账户 B 的余额仍然会反映转账后的状态。

隔离级别

1. Read Uncommitted (读未提交)

  • 特点: 事务可以读取到其他未提交事务的数据。
  • 问题:
    • 脏读 (Dirty Read): 事务 A 读取了事务 B 未提交的数据，如果事务 B 回滚，事务 A 读取的数据就是无效的。
  • 应用场景: 几乎没有实际应用场景，因为脏读会导致数据严重不一致。

2. Read Committed (读已提交)

  • 特点: 事务只能读取到其他已提交事务的数据。
  • 问题:
    • 不可重复读 (Non-repeatable Read): 同一个事务内多次读取同一数据，读取到的结果可能不同，因为其他事务可能已经修改并提交了该数据。
  • 应用场景: 大多数数据库默认的隔离级别，可以避免脏读，但存在不可重复读的问题。

3. Repeatable Read (可重复读)

  • 特点: 同一个事务内多次读取同一数据，读取到的结果相同，即使其他事务已经修改并提交了该数据。
  • 问题:
    • 幻读 (Phantom Read): 事务 A 读取某个范围的数据，事务 B 在该范围内插入或删除了数据，导致事务 A 再次读取时，发现结果集发生了变化。
  • 应用场景: MySQL InnoDB 引擎默认的隔离级别，可以避免脏读和不可重复读，但存在幻读的问题。

4. Serializable (串行化)

  • 特点: 所有事务串行执行，相当于只有一个事务在执行，避免了所有并发问题。
  • 问题: 并发性能极低，因为所有事务都需要排队执行。
  • 应用场景: 对数据一致性要求极高的场景，例如金融交易。

UNDO Log

  • 针对单个行的undo log： 对于数据库的每一行数据，UNDO日志会记录对该行数据的每一次修改之前的值，比如针对某行数据的UPDATE操作，会记录修改之前该行数据的完整内容。这些UNDO日志条目会通过类似链表的方式连接起来。 新的修改产生新的UNDO日志条目，指向之前的UNDO日志条目，形成一个历史链。
  • 针对整个事务的undo log： 对于整个事务来说，事务开始的时候，会记录一个特殊的UNDO日志条目，这个条目可以理解为整个事务的链表头。 事务中，每一个操作产生的UNDO日志条目会链接到这个链表上。 这样，就可以通过这个链表，按照操作相反的顺序，撤销事务中所有的修改。

例如：

  1. 对于 insert 操作:
    • Undo Log 会记录下如何撤销 insert 操作的信息。 通常，这会包括删除刚刚创建的行的必要信息（例如：表名、行的唯一标识符）。
    • Undo Log 实际上存储的是删除该行的操作。
  2. 对于 update 操作:
    • Undo Log 会记录被修改行的原始值（修改前的完整数据）。
    • Undo Log 会记录修改发生所在的表、行标识符以及被修改的字段。
    • Undo Log 中会包含指向前一个版本 Undo Log 的指针，形成一个链。
  3. 事务的链式结构:
    • 事务开始时，会创建一个 “事务开始” Undo Log 条目。
    • 该事务内的所有 CREATE 和 CHANGE 操作的 Undo Log 条目都会链接到这个 “事务开始” 条目，形成一个链。 这个链保证了可以按照相反的顺序撤销事务中的所有操作。

Undo Log

Undo Log 刷盘

在 MySQL 中，UNDO LOG 默认是存储在 InnoDB 的表空间中的，具体来说，它存储在 Undo Tablespace 中。UNDO LOG 并不是直接存储在内存中，而是会定期刷盘到磁盘上，以确保事务的持久性和一致性。

在 MySQL 5.7 及更高版本中，UNDO LOG 默认存储在独立的 Undo Tablespace 中（而不是在系统表空间中），并且这些 Undo Tablespace 文件会定期刷盘到磁盘上。

内存中的 UNDO LOG 如何清理？

内存中的 UNDO LOG 主要通过以下机制进行清理：

  1. 事务提交或回滚：当一个事务提交或回滚时，相关的 UNDO LOG 会被标记为可清理状态。事务提交后，UNDO LOG 不再需要用于回滚，因此可以被清理。
  2. Purge 线程：MySQL 有一个后台的 Purge 线程，负责清理不再需要的 UNDO LOG。Purge 线程会定期扫描 UNDO LOG，并清理那些已经不再被任何活动事务引用的 UNDO LOG 记录。
  3. Undo Tablespace 回收：当 UNDO LOG 被清理后，相关的空间会被回收，以便后续的事务可以复用这些空间。MySQL 会自动管理 Undo Tablespace 的大小，确保不会无限增长。
  4. Undo Log 的轮转：MySQL 会定期轮转 UNDO LOG，即当某个 Undo Tablespace 中的 UNDO LOG 记录不再需要时，MySQL 会将其标记为可重用，并在需要时覆盖这些空间。

redo log 和 binlog

事务开始时，记录对数据库的修改，同时写入redo log和Undo log，如果发生崩溃，可通过redo log重放恢复到崩溃前状态

redo log 和 binlog 均需要持久化到磁盘

redo log在磁盘上是顺序写，效率高于修改数据后直接写磁盘时的随机写；

因此，修改数据直接修改位于Buffer Pool中的脏页，通过追加写redo log刷盘

redo log非直接写入磁盘，相似的，redolog先写入redo log buffer，在以下时机刷入磁盘：

  • MySQL正常关闭时刷入磁盘
  • 当redo log buffer中记录量大于redo log buffer总容量的一半时刷入磁盘
  • Innodb后台线程每1秒执行刷入磁盘
  • 事务提交时刷入磁盘（参数innodb_flush_log_at_trx_commit控制）

文件系统中存在两个redo log file，redo log buffer刷盘时，环形写入redo log file0和redo log file1；

check point表示buffer pool中脏页已刷盘，不需要重放的数据。如果write pos追上check point，则MySQL会强制暂停开始将buffer pool脏页刷盘；

事务每条语句会产生一条 binlog 记录到 binlog buffer；当事务提交时，写 binlog buffer 到磁盘；

主索引与辅助索引

是否需要建立索引以及在哪些列上建立索引通常由程序员手动指定。虽然一些数据库系统（如 MySQL）可能自动为主键和唯一键创建索引，但其他索引需要开发人员根据实际查询需求来手动设计和创建。

当为其他列创建索引时，数据库会额外创建一棵新的 B+ 树，用于存储该列的索引结构。

创建表时，主索引和辅助索引的结构：

  • 在使用 InnoDB 存储引擎时：
    • 主索引实际上就是数据本身，数据按照主键的顺序存储。
    • 辅助索引是单独的结构，它存储了对应的列值（即索引列值）和主键值的映射关系。
  • 辅助索引并不是单独存储“完整数据行”，它仅包含辅助索引列和主键值。

  • 如果 WHERE 子句使用主键（如 id = 10），直接通过主索引找到数据行，无需回表，查询效率最高。
  • 如果 WHERE 子句使用的是辅助索引列（如 name = 'Alice'），需要回表查找完整数据行，效率相对稍低。
  • 如果查询只涉及辅助索引的列（如 SELECT name FROM students WHERE name = 'Alice'），MySQL 可能会 覆盖索引，直接从辅助索引中返回结果，无需回表。

联合索引与优化

联合索引核心：

（a,b,c）a全局有序，在a相同的情况下b有序，ab相同情况下c有序

联合索引将多个列的值按照一定的顺序拼接在一起，作为 B+ 树中每个节点的 key。树节点按照索引列的顺序存储，先比较第一个列（即主列）的值，如果相同，再比较第二个列的值，以此类推。

索引列的顺序很重要，应按照查询条件的常用模式设计，以充分利用左前缀原则。

当b+树的数据项是复合的数据结构，比如(name,age,sex)的时候，b+数是按照从左到右的顺序来建立搜索树的，比如当(张三,20,F)这样的数据来检索的时候，b+树会优先比较name来确定下一步的所搜方向，如果name相同再依次比较age和sex，最后得到检索的数据；但当(20,F)这样的没有name的数据来的时候，b+树就不知道下一步该查哪个节点，因为建立搜索树的时候name就是第一个比较因子，必须要先根据name来搜索才能知道下一步去哪里查询。比如当(张三,F)这样的数据来检索时，b+树可以用name来指定搜索方向，但下一个字段age的缺失，所以只能把名字等于张三的数据都找到，然后再匹配性别是F的数据了， 这个是非常重要的性质，即索引的最左匹配特性。

前缀索引

对于 BLOB、TEXT 和 VARCHAR 类型的列，使用前缀索引可以降低索引的存储空间，同时提升查询效率。

  • 全列索引（索引整个 email 列）可能占用大量空间，特别是对于长字符串。

  • 前缀索引通过截取前 N 个字符，既能节省空间又能加速匹配。

  • 前缀长度应根据选择性确定，避免索引过短导致大量重复值。

MySQL 不使用索引的情况：

  • 索引列被计算或函数包裹。
  • 复合索引不符合最左前缀原则。
  • 存在范围查询阻断索引匹配。
  • 列的区分度过低或隐式类型转换。

索引的使用场景

  • 对于非常小的表、大部分情况下简单的全表扫描比建立索引更高效。
  • 对于中到大型的表，索引就非常有效。
  • 但是对于特大型的表，建立和维护索引的代价将会随之增长。这种情况下，需要用到一种技术可以直接区分出需要查询的一组数据，而不是一条记录一条记录地匹配，例如可以使用分区技术。

自适应哈希索引 (Adaptive Hash Index, AHI)

  1. 监控热点页: InnoDB 引擎会监控索引页的访问频率。当某个索引页被频繁访问，达到预设阈值后，会被认为是热点页。

  3. 加速查询: 当执行查询时，InnoDB 引擎会先查找哈希索引。如果找到匹配项，可以直接定位到对应的索引页，避免了 B-tree 索引的多次查找，从而提升查询效率。

Undo Buffer && Undo Tablespace

Undo Log 记录了事务中每一个数据修改操作的反向操作，例如：

  • 插入操作的 Undo Log 记录了删除相应记录的操作。
  • 更新操作的 Undo Log 记录了将数据恢复到旧值的操作。
  • 删除操作的 Undo Log 记录了插入被删除记录的操作。

Undo Log 主要用于事务回滚和实现 MVCC

读取一致性 (MVCC)： 即使事务 A 已经提交，其他未提交的事务或使用了快照读的事务仍然可能需要访问事务 A 修改之前的数据版本。Undo Log 中保留的事务 A 的记录可以用来构建这些旧版本的数据，保证读取一致性。

Undo Log 存储在 Undo Tablespace 中，可以配置为独立的表空间或 InnoDB 系统表空间的一部分

Redo Log Buffer && Redo Log Files

Redo Log 记录了数据库的物理修改，例如：

  • 在哪个数据页修改了哪个字节，修改后的值是什么。

当事务修改数据时，先将修改记录写入 Redo Log Buffer, 在事务提交时，将 Redo Log Buffer 中的内容刷新到 Redo Log Files, 数据库崩溃恢复/内存掉电数据清空时，会读取 Redo Log Files，重新执行其中的修改操作，将数据库恢复到崩溃前的最新状态。

Redo Log 的作用是保证已提交事务的持久性，而不是未提交事务的恢复。

Redo Log 存储在 Redo Log Files 中，通常是两个或多个循环使用的文件。

MySQL主从复制

MySQL的主从复制为BinLog

主要涉及三个线程: binlog 线程、I/O 线程和 SQL 线程。

  • binlog 线程 : 负责将主服务器上的数据更改写入二进制日志中。
  • I/O 线程 : 负责从主服务器上读取二进制日志，并写入从服务器的中继日志中。
  • SQL 线程 : 负责读取中继日志并重放其中的 SQL 语句。

流程

  1. 主数据库接收到一个写操作（如 INSERT、UPDATE、DELETE）时，会将这个操作记录到二进制日志（Binary Log）中，将数据修改的操作按顺序记录下来。
  2. 从数据库 IO 线程会自动连接主数据库，从二进制日志中读取操作信息，记录到中继日志（Relay Log）中。
  3. 从数据库的 SQL 线程会定期从中继日志中获取同步数据，写入到从数据库中。

Bin Log 日志格式

Binary Log 二级制日志，它总共有以下三种格式（不同的日志格式决定了不同的主从同步效果）：

  1. STATEMENT 格式（语句模式，出现在 MySQL 5.1 之前）：在这种格式下，binlog 记录的是执行的 SQL 语句的文本。
    1. 优点：日志文件通常较小，复制效率较高。
    2. 缺点：在某些情况下，由于数据库环境的差异（如表结构、字符集等），在从服务器上重放这些 SQL 语句可能会导致不一致的结果。例如，获取当前时间的函数或存储过程等，可能会导致数据不一致。
  2. ROW 格式（行模式，诞生于 MySQL 5.1）：在这种格式下，binlog 记录的是每一行数据更改的具体内容。
    1. 优点：能够精确地记录数据的变化，避免了 STATEMENT 格式中的环境依赖问题，提供了更强的一致性保证。
    2. 缺点：日志文件可能会比 STATEMENT 格式大，因为记录了每一行的详细变化。此外，ROW 格式的日志在进行大量数据更新时可能会导致更高的 I/O 开销。
  3. MIXED 格式（混合模式）：在这种格式下，binlog 可以根据具体的 SQL 语句和操作自动选择使用 STATEMENT 或 ROW 格式。
    1. 优点：结合了 STATEMENT 和 ROW 格式的优点，能够在保证一致性的同时尽可能地优化日志大小和复制性能。
    2. 缺点：由于混合使用了两种格式，可能需要更复杂的管理和监控。在某些特定情况下，MIXED 格式可能无法达到最优的性能或一致性。

主从复制模式

MySQL 中主要有以下两种主从复制的模式，分别是异步复制和半同步复制。

  1. 异步复制：MySQL 主从复制中最常见和默认的模式。在异步复制模式中，主服务器将数据修改操作记录到二进制日志（Binary Log）中，并由从服务器的IO线程读取主服务器日志。从服务器接收到二进制日志后，会异步地应用这些日志进行数据复制。
    1. 优点：它的优点是及时响应给使用者，主服务器不会受到从服务器的影响而等待确认，可以提高主服务器的性能。
    2. 缺点：由于是异步复制，可能存在数据传输的延迟，且从服务器上的复制过程是不可靠的。如果主服务器故障，尚未应用到从服务器的数据可能会丢失。
  2. 半同步复制：半同步复制是 MySQL 主从复制中的一种增强模式。在半同步复制模式中，主服务器将数据修改操作记录到二进制日志，并等待至少一个从服务器确认已接收到并应用了这些日志后才继续执行后续操作。
    1. 优点：可以提供更高的数据一致性和可靠性，确保至少一个从服务器与主服务器保持同步。如果主服务器故障，已经确认接收并应用到从服务器的数据不会丢失。
    2. 缺点：由于半同步复制需要等待从服务器的确认，因此相对于异步复制，会增加一定的延迟，可能会影响主服务器的性能。

如果对数据一致性和可靠性要求较高，可以考虑使用半同步复制；如果对延迟和主服务器性能要求较高，可以继续使用异步复制，根据实际需求调整复制模式。

错误用法：from_unixtime(create_time) = '2014-05-29'

  • 这个查询会使用from_unixtime函数对create_time进行转换，然后与字符串'2014-05-29'比较。
  • 问题是，在这种情况下，create_time字段的索引将无法使用，因为from_unixtime(create_time)会使得数据库需要对每个create_time值都进行函数计算，无法直接通过索引定位到对应的记录。
  • 结果是，查询性能会下降，数据库可能需要全表扫描。

优化用法：create_time = unix_timestamp('2014-05-29')

  • 在这种情况下，查询直接使用create_time字段的原始值进行比较，不需要任何计算或函数调用。
  • 因为create_time列上的索引可以直接用于定位记录，查询性能大大提高。

Redis持久化

Redis 的数据持久化机制允许将内存中的数据保存到磁盘文件，以便在服务重启或宕机后可以恢复数据。主要有以下两种机制：

RDB（Redis DataBase）持久化

RDB 是 Redis 的快照方式，定期将内存中的数据以二进制文件的形式保存到磁盘。

工作原理：

  • Redis 会通过 SAVE 或 BGSAVE 命令生成 RDB 文件。
    • SAVE 命令：
      • 在主线程中完成快照保存操作，阻塞 Redis 的读写。
    • BGSAVE 命令：
      • Redis 创建一个子进程，子进程负责将数据写入 RDB 文件，主线程继续处理客户端请求。
      • 子进程生成快照时使用写时复制（Copy-On-Write, COW）技术，保证数据一致性。

触发方式：

  • 自动触发：通过配置文件中的 save 选项设置定期触发规则。例如：

  • 手动触发：执行 SAVE 或 BGSAVE 命令。

读写影响：

  • SAVE 命令：阻塞主线程，影响读写性能，适合非高负载场景。
  • BGSAVE 命令：使用子进程完成，但可能增加系统 I/O 压力，特别是数据量较大时。

AOF（Append Only File）持久化

AOF 是 Redis 的日志方式，将每次写操作记录到日志文件中，可以通过重放日志恢复数据。

工作原理：

  • Redis 将每次写操作的命令追加到 AOF 文件末尾。
  • 支持三种同步策略（通过 appendfsync 参数配置）：
    1. always：每次写操作都同步到磁盘，最安全但性能较差。
    2. everysec：每秒同步一次，默认设置，兼顾性能与数据安全。
    3. no：由操作系统决定何时同步，性能最好，但可能丢失数据。

触发重写：

  • AOF 文件会不断增长，因此需要定期重写（rewrite）以压缩日志。
  • Redis 通过 BGREWRITEAOF 命令创建子进程完成重写，类似 BGSAVE。

读写影响：

  • 追加模式：性能影响较小，因写操作只需追加日志。
  • 重写操作：由子进程完成，主线程继续处理请求，可能增加 I/O 压力。

Redis 淘汰机制

Redis 过期键的删除策略

Redis 主要有两种策略来删除过期键：

  • 惰性删除（Lazy Deletion）：当一个键被访问时，Redis 会检查该键是否已经过期。如果已过期，Redis 会删除该键，并返回相应的空值。这种方式只会在键被访问时检查过期情况，并删除过期键，可能会导致一些过期键在一段时间内仍然占用内存。
  • 定期删除（Periodic Deletion）：Redis 会定期随机检查一部分过期的键，并将其删除。默认情况下，每 100 毫秒 Redis 会随机检查 10 个过期的键进行删除。这种方式不会让过期键永久占用内存，但仍可能存在一些过期键在下一次检查之前不会被删除。

  • 实现：Redis 使用一种近似 LRU 算法，通过采样法（默认 5 个键）来选择被淘汰的键。

Redis 哨兵的选举机制是什么样的？

Redis 哨兵（Sentinel）是 Redis 集群中负责监控、通知、自动故障转移和服务发现的组件。它的选举机制是确保在主节点（Master）故障时，能够自动选举一个新的主节点，并促使系统恢复正常运行。

  • 哨兵的选举机制是什么样的？

哨兵的选举机制其实很简单，就是一个Raft选举算法： 选举的票数大于等于num(sentinels)/2+1时，将成为领导者，如果没有超过，继续选举

Raft算法可以参看这篇文章分布式算法 - Raft算法

  • 任何一个想成为 主节点 的节点，要满足两个条件：
    • 第一，拿到半数以上的赞成票；
    • 第二，拿到的票数同时还需要大于等于哨兵配置文件中的 quorum 值。

以 3 个哨兵为例，假设此时的 quorum 设置为 2，那么，任何一个想成为主节点 只要拿到 2 张赞成票，就可以了。

Lua脚本 与 Redis事务

Lua脚本：

  • Redis 在执行 Lua 脚本时，保证脚本中的所有命令在一个原子操作内完成。即使脚本包含多个 Redis 命令，这些命令也不会被其他客户端的命令插入。
  • Redis 使用单线程模型，Lua 脚本执行期间不会有其他命令被执行，因此具备天然的原子性。

事务（MULTI/EXEC）：

  • Redis 事务中的命令从 MULTI 到 EXEC 的提交过程被称为事务块。但 Redis 不保证事务中的命令是完全原子的。
  • 如果事务中的某些命令失败（例如操作数据类型错误），事务中其他命令仍会执行。
  • 事务本质上是批量发送命令，但它不提供类似 Lua 的原子性保障。

“事务本质上是批量发送命令”，这意味着如果多个客户端向redis服务端发送事务块，由于一个redis服务端执行命令是单线程的，有很大可能这多个客户端的事务块中的命令是交错执行的，没有保证原子性

一致哈希算法是对 2^32 进行取模运算，是一个固定的值。

对存储节点进行哈希计算，也就是对存储节点做哈希映射，比如根据节点的 IP 地址进行哈希；

当对数据进行存储或访问时，对数据进行哈希映射；

一致性哈希是指将「存储节点」和「数据」都映射到一个首尾相连的哈希环上。
