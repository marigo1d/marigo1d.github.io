LIKE 运算符:

用于模式匹配。

SQL LIKE 运算符用于模式匹配，允许你搜索包含特定字符序列的数据。LIKE 运算符通常与通配符一起使用，以创建更灵活的搜索模式。以下是常用的 LIKE 通配符：

  • % (百分号): 匹配任意字符序列（包括空序列）。它可以用于匹配字符串的开头、结尾或中间的任意部分。

    • 示例:
      • LIKE 'A%': 匹配以 “A” 开头的任何字符串，例如 “Apple”、”Airplane”、”A”。
      • LIKE '%e': 匹配以 “e” 结尾的任何字符串，例如 “Apple”、”Orange”、”e”。
      • LIKE '%app%': 匹配包含 “app” 的任何字符串，例如 “Apple”、”Application”、”Happy”。

  • _ (下划线): 匹配任何单个字符。

    • 示例:
      • LIKE 'A_': 匹配以 “A” 开头且后面跟着一个任意字符的字符串，例如 “Ab”、”Ac”、”A1”。
      • LIKE '_pple': 匹配以任意字符开头且后面跟着 “pple” 的字符串，例如 “Apple”、”ipple”。
      • LIKE 'a__le': 匹配以 “a” 开头，后面跟着两个任意字符，再跟着 “le” 的字符串，例如 “apple”、”azble”。

  • [] (方括号): 匹配指定集合中的任何单个字符。

    • 示例:
      • LIKE '[ABC]%': 匹配以 “A”、”B” 或 “C” 开头的任何字符串。
      • LIKE 'h[aeiou]t': 匹配 “hat”、”het”、”hit”、”hot” 或 “hut”。
      • LIKE '[A-Z]%': 匹配以任何大写字母开头的字符串。

  • [^] (方括号加脱字符): 匹配不在指定集合中的任何单个字符。

    • 示例:
      • LIKE '[^ABC]%': 匹配不以 “A”、”B” 或 “C” 开头的任何字符串。
      • LIKE 'h[^aeiou]t': 匹配以 “h” 开头，第二个字符不是元音字母，以 “t” 结尾的字符串。

数据库事务的 ACID 特性

ACID 是数据库事务的四个重要特性，它们保证了事务的可靠性和一致性。

1. 原子性 (Atomicity)

  • 定义: 一个事务的所有操作要么全部成功执行，要么全部不执行，不会出现部分成功部分失败的情况。
  • 作用: 保证数据库状态的一致性，即使事务执行过程中发生错误，数据库也能回滚到事务开始前的状态，就像什么都没发生过一样。
  • 举例: 转账操作需要从账户 A 扣款并向账户 B 加款，这两个操作必须作为一个整体执行，要么都成功，要么都失败。如果只扣款成功而加款失败，就会导致数据不一致。

2. 一致性 (Consistency)

  • 定义: 事务执行前后，数据库必须保持一致的状态，满足所有预定义的约束和规则。
  • 作用: 保证数据库数据的完整性和正确性，防止数据出现逻辑上的错误。
  • 举例: 银行账户的余额不能小于零。如果一个事务导致账户余额变为负数，则该事务违反了一致性约束，数据库会回滚事务，保持数据一致性。

3. 隔离性 (Isolation)

  • 定义: 多个事务并发执行时，彼此之间相互隔离，互不干扰，就像每个事务都是独立执行的一样。
  • 作用: 防止并发事务之间的数据冲突，保证每个事务都能看到一致的数据。
  • 举例: 用户 A 和用户 B 同时读取账户 C 的余额，然后分别进行转账操作。隔离性保证了用户 A 和用户 B 的操作不会相互影响，每个用户都能看到正确的账户余额。

4. 持久性 (Durability)

  • 定义: 一旦事务提交成功，其对数据库的修改将永久保存，即使数据库发生故障也不会丢失。
  • 作用: 保证数据的可靠性和安全性，即使系统崩溃，数据也能恢复到最新状态。
  • 举例: 用户 A 成功转账后，即使数据库突然断电，转账操作的结果也不会丢失，数据库重启后，账户 A 和账户 B 的余额仍然会反映转账后的状态。

隔离级别

1. Read Uncommitted (读未提交)

  • 特点: 事务可以读取到其他未提交事务的数据。
  • 问题:
    • 脏读 (Dirty Read): 事务 A 读取了事务 B 未提交的数据，如果事务 B 回滚，事务 A 读取的数据就是无效的。
  • 应用场景: 几乎没有实际应用场景，因为脏读会导致数据严重不一致。

2. Read Committed (读已提交)

  • 特点: 事务只能读取到其他已提交事务的数据。
  • 问题:
    • 不可重复读 (Non-repeatable Read): 同一个事务内多次读取同一数据，读取到的结果可能不同，因为其他事务可能已经修改并提交了该数据。
  • 应用场景: 大多数数据库默认的隔离级别，可以避免脏读，但存在不可重复读的问题。

3. Repeatable Read (可重复读)

  • 特点: 同一个事务内多次读取同一数据，读取到的结果相同，即使其他事务已经修改并提交了该数据。
  • 问题:
    • 幻读 (Phantom Read): 事务 A 读取某个范围的数据，事务 B 在该范围内插入或删除了数据，导致事务 A 再次读取时，发现结果集发生了变化。
  • 应用场景: MySQL InnoDB 引擎默认的隔离级别，可以避免脏读和不可重复读，但存在幻读的问题。

4. Serializable (串行化)

  • 特点: 所有事务串行执行，相当于只有一个事务在执行，避免了所有并发问题。
  • 问题: 并发性能极低，因为所有事务都需要排队执行。
  • 应用场景: 对数据一致性要求极高的场景，例如金融交易。

主索引与辅助索引

是否需要建立索引以及在哪些列上建立索引通常由程序员手动指定。虽然一些数据库系统（如 MySQL）可能自动为主键和唯一键创建索引，但其他索引需要开发人员根据实际查询需求来手动设计和创建。

当为其他列创建索引时，数据库会额外创建一棵新的 B+ 树，用于存储该列的索引结构。

创建表时，主索引和辅助索引的结构：

  • 在使用 InnoDB 存储引擎时：
    • 主索引实际上就是数据本身，数据按照主键的顺序存储。
    • 辅助索引是单独的结构，它存储了对应的列值（即索引列值）和主键值的映射关系。
  • 辅助索引并不是单独存储“完整数据行”，它仅包含辅助索引列和主键值。

这种过程叫做 回表操作，因为查询先通过辅助索引，再“回到”主索引获取数据。

主索引 vs 辅助索引的查询效率

  • 如果 WHERE 子句使用主键（如 id = 10），直接通过主索引找到数据行，无需回表，查询效率最高。
  • 如果 WHERE 子句使用的是辅助索引列（如 name = 'Alice'），需要回表查找完整数据行，效率相对稍低。
  • 如果查询只涉及辅助索引的列（如 SELECT name FROM students WHERE name = 'Alice'），MySQL 可能会 覆盖索引，直接从辅助索引中返回结果，无需回表。

多列索引与优化

多列索引将多个列的值按照一定的顺序拼接在一起，作为 B+ 树中每个节点的 key。树节点按照索引列的顺序存储，先比较第一个列（即主列）的值，如果相同，再比较第二个列的值，以此类推。

索引列的顺序很重要，应按照查询条件的常用模式设计，以充分利用左前缀原则。

当b+树的数据项是复合的数据结构，比如(name,age,sex)的时候，b+数是按照从左到右的顺序来建立搜索树的，比如当(张三,20,F)这样的数据来检索的时候，b+树会优先比较name来确定下一步的所搜方向，如果name相同再依次比较age和sex，最后得到检索的数据；但当(20,F)这样的没有name的数据来的时候，b+树就不知道下一步该查哪个节点，因为建立搜索树的时候name就是第一个比较因子，必须要先根据name来搜索才能知道下一步去哪里查询。比如当(张三,F)这样的数据来检索时，b+树可以用name来指定搜索方向，但下一个字段age的缺失，所以只能把名字等于张三的数据都找到，然后再匹配性别是F的数据了， 这个是非常重要的性质，即索引的最左匹配特性。

为什么创建多列索引比单列索引性能更高？

  • 单列索引的劣势：
    • 如果单独对 customer_id 和 staff_id 分别创建索引，查询条件中既有 customer_id 又有 staff_id 时，MySQL 会分别在两个索引中查找符合条件的行，然后取交集。这种过程需要更多的计算。
    • 单列索引不能直接优化多列组合查询的场景。
  • 多列索引的优势：
    • 多列索引可以一次性通过 (customer_id, staff_id) 的组合值查找到符合条件的行，无需取交集。
    • 索引是有序的，因此范围查询（如 customer_id = 101 AND staff_id BETWEEN 1 AND 2）也能快速执行。

前缀索引

对于 BLOB、TEXT 和 VARCHAR 类型的列，使用前缀索引可以降低索引的存储空间，同时提升查询效率。

  • 全列索引（索引整个 email 列）可能占用大量空间，特别是对于长字符串。

  • 前缀索引通过截取前 N 个字符，既能节省空间又能加速匹配。

  • 前缀长度应根据选择性确定，避免索引过短导致大量重复值。

MySQL 不使用索引的情况：

  • 索引列被计算或函数包裹。
  • 复合索引不符合最左前缀原则。
  • 存在范围查询阻断索引匹配。
  • 列的区分度过低或隐式类型转换。

索引的使用场景

  • 对于非常小的表、大部分情况下简单的全表扫描比建立索引更高效。
  • 对于中到大型的表，索引就非常有效。
  • 但是对于特大型的表，建立和维护索引的代价将会随之增长。这种情况下，需要用到一种技术可以直接区分出需要查询的一组数据，而不是一条记录一条记录地匹配，例如可以使用分区技术。

自适应哈希索引 (Adaptive Hash Index, AHI)

  1. 监控热点页: InnoDB 引擎会监控索引页的访问频率。当某个索引页被频繁访问，达到预设阈值后，会被认为是热点页。
  2. 创建哈希索引: InnoDB 引擎会自动为热点页创建哈希索引，并将哈希索引存储在内存的哈希表中。
  3. 加速查询: 当执行查询时，InnoDB 引擎会先查找哈希索引。如果找到匹配项，可以直接定位到对应的索引页，避免了 B-tree 索引的多次查找，从而提升查询效率。

Undo Buffer && Undo Tablespace

Undo Log 记录了事务中每一个数据修改操作的反向操作，例如：

  • 插入操作的 Undo Log 记录了删除相应记录的操作。
  • 更新操作的 Undo Log 记录了将数据恢复到旧值的操作。
  • 删除操作的 Undo Log 记录了插入被删除记录的操作。

Undo Log 主要用于事务回滚和实现 MVCC

读取一致性 (MVCC)： 即使事务 A 已经提交，其他未提交的事务或使用了快照读的事务仍然可能需要访问事务 A 修改之前的数据版本。Undo Log 中保留的事务 A 的记录可以用来构建这些旧版本的数据，保证读取一致性。

Undo Log 存储在 Undo Tablespace 中，可以配置为独立的表空间或 InnoDB 系统表空间的一部分

Redo Log Buffer && Redo Log Files

Redo Log 记录了数据库的物理修改，例如：

  • 在哪个数据页修改了哪个字节，修改后的值是什么。

当事务修改数据时，先将修改记录写入 Redo Log Buffer, 在事务提交时，将 Redo Log Buffer 中的内容刷新到 Redo Log Files, 数据库崩溃恢复/内存掉电数据清空时，会读取 Redo Log Files，重新执行其中的修改操作，将数据库恢复到崩溃前的最新状态。

Redo Log 的作用是保证已提交事务的持久性，而不是未提交事务的恢复。

Redo Log 存储在 Redo Log Files 中，通常是两个或多个循环使用的文件。

Redis持久化

Redis 的数据持久化机制允许将内存中的数据保存到磁盘文件，以便在服务重启或宕机后可以恢复数据。主要有以下两种机制：

RDB（Redis DataBase）持久化

RDB 是 Redis 的快照方式，定期将内存中的数据以二进制文件的形式保存到磁盘。

工作原理：

  • Redis 会通过 SAVE 或 BGSAVE 命令生成 RDB 文件。
    • SAVE 命令：
      • 在主线程中完成快照保存操作，阻塞 Redis 的读写。
    • BGSAVE 命令：
      • Redis 创建一个子进程，子进程负责将数据写入 RDB 文件，主线程继续处理客户端请求。
      • 子进程生成快照时使用写时复制（Copy-On-Write, COW）技术，保证数据一致性。

触发方式：

  • 自动触发：通过配置文件中的 save 选项设置定期触发规则。例如：

  • 手动触发：执行 SAVE 或 BGSAVE 命令。

读写影响：

  • SAVE 命令：阻塞主线程，影响读写性能，适合非高负载场景。
  • BGSAVE 命令：使用子进程完成，但可能增加系统 I/O 压力，特别是数据量较大时。

AOF（Append Only File）持久化

AOF 是 Redis 的日志方式，将每次写操作记录到日志文件中，可以通过重放日志恢复数据。

工作原理：

  • Redis 将每次写操作的命令追加到 AOF 文件末尾。
  • 支持三种同步策略（通过 appendfsync 参数配置）：
    1. always：每次写操作都同步到磁盘，最安全但性能较差。
    2. everysec：每秒同步一次，默认设置，兼顾性能与数据安全。
    3. no：由操作系统决定何时同步，性能最好，但可能丢失数据。

触发重写：

  • AOF 文件会不断增长，因此需要定期重写（rewrite）以压缩日志。
  • Redis 通过 BGREWRITEAOF 命令创建子进程完成重写，类似 BGSAVE。

读写影响：

  • 追加模式：性能影响较小，因写操作只需追加日志。
  • 重写操作：由子进程完成，主线程继续处理请求，可能增加 I/O 压力。

Redis 淘汰机制

通过参数 maxmemory 设置 Redis 实例的内存上限。可以在配置文件中设置，也可以通过运行时命令动态配置：

Redis 淘汰策略

通过参数 maxmemory-policy 配置 Redis 的淘汰策略。主要分为以下几种：

（1）noeviction

  • 描述：当内存不足时，拒绝写入新数据。
  • 适用场景：数据不可丢失的场景，常用于缓存外的其他用途。
  • 行为：客户端会收到 OOM 错误。

（2）allkeys-lru

  • 描述：在所有键中，根据最近最少使用（LRU）原则淘汰。
  • 适用场景：适合纯缓存场景，优先淘汰“冷”数据。

（3）volatile-lru

  • 描述：仅在设置了过期时间的键中，根据 LRU 原则淘汰。
  • 适用场景：当需要保留永久性数据，仅清除临时性数据。

（4）allkeys-random

  • 描述：在所有键中随机淘汰数据。
  • 适用场景：简单且无需明确淘汰规则的场景。

（5）volatile-random

  • 描述：仅在设置了过期时间的键中随机淘汰数据。
  • 适用场景：用于保留永久性数据并随机清除过期键。

（6）volatile-ttl

  • 描述：仅在设置了过期时间的键中，优先淘汰剩余生存时间（TTL）最短的键。
  • 适用场景：适用于希望根据键的“即将过期”程度来清理数据的场景。

（7）allkeys-lfu

  • 描述：在所有键中，根据最少使用频率（LFU）淘汰。
  • 适用场景：适合需要保留高频使用数据的场景。

（8）volatile-lfu

  • 描述：仅在设置了过期时间的键中，根据最少使用频率（LFU）淘汰。
  • 适用场景：结合 LFU 和过期时间淘汰策略的场景。

LRU 与 LFU 机制

（1）LRU（Least Recently Used）

  • 原理：淘汰最久未被访问的键。
  • 实现：Redis 使用一种近似 LRU 算法，通过采样法（默认 5 个键）来选择被淘汰的键。

（2）LFU（Least Frequently Used）

  • 原理：淘汰访问频率最低的键。
  • 实现：Redis 通过维护一个计数器（counter）跟踪每个键的访问频率，采用递减机制避免计数器无限增长。

Lua脚本 与 Redis事务

Lua脚本：

  • Redis 在执行 Lua 脚本时，保证脚本中的所有命令在一个原子操作内完成。即使脚本包含多个 Redis 命令，这些命令也不会被其他客户端的命令插入。
  • Redis 使用单线程模型，Lua 脚本执行期间不会有其他命令被执行，因此具备天然的原子性。

事务（MULTI/EXEC）：

  • Redis 事务中的命令从 MULTI 到 EXEC 的提交过程被称为事务块。但 Redis 不保证事务中的命令是完全原子的。
  • 如果事务中的某些命令失败（例如操作数据类型错误），事务中其他命令仍会执行。
  • 事务本质上是批量发送命令，但它不提供类似 Lua 的原子性保障。

“事务本质上是批量发送命令”，这意味着如果多个客户端向redis服务端发送事务块，由于一个redis服务端执行命令是单线程的，有很大可能这多个客户端的事务块中的命令是交错执行的，没有保证原子性
