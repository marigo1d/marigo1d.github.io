Skill

MCP connects Cluade to data; Skills teach Claude what to do with that data

每个skill对应一个skill文档，通过类似于tool的方式暴露skill元数据给llm，由llm决定加载所需skill的完整文档（完整prompt）

可用理解为对command的进一步分层封装，将多个command的元信息暴露给

skill文档中可通过自然语言描述的方式引入reference（例如，当提到“钱，预算”时触发：读取“财务说明.md”），追加reference内容到上下文；

skill文档中可通过自然语言描述的方式引入script（例如，当提到“上传”时触发：运行upload.py脚本上传文件内容），追加**脚本执行状态（不读取脚本）**到上下文；

特性

  • 渐进式披露
  • reference按需加载
  • script选择性触发

设计

接口暴露

Tool和Skill的使用均依赖于对应元数据对LLM的暴露，

通过元数据的描述为LLM选择调用哪个tool/skill提供基础

可以理解为后端暴露接口需向前端提供说明文档


引用

引用注重于为上下文追加所需信息

核心：仅为llm提供数据的引用，当需要时进行进一步加载

类似于惰性加载

包括

  • 告知llm文件目录格式，当llm需要某文件详细内容时追加至上下文
  • 告知llm可用skill，当llm需要时追加至上下文

调用

调用注重于直接获取执行结果而不关心执行过程

包括

  • 告知llm可用脚本，当llm需要某数据时执行脚本并追加脚本获得数据
  • 告知llm可用脚本，当llm需要某动作时执行脚本并追加脚本执行结果

触发

  • 正反例
  • 精确的关键词匹配

Fin

个人在学习时也看到了很多关于skill和mcp的争论，其实感觉由于现在agent技术的蓬勃发展，很多术语可能也是从不同角度对同一个事物的描述，顺便记录一下自己对见到的一些观点的看法吧


skill = sop?

  • skill确实可以做sop的事情（通过自然语言描述软编码实现），感觉更像是模型能力提升后的进一步发展
  • 个人认为实际上如果是标准化规范化的场景，更应该做成coze的方式通过程序化硬编码的方式规范状态机的状态与转移

skill是为了省上下文？

  • 个人认为确实是，进一步的说为什么2026年了还需要节省上下文？

ref: https://github.com/Lianues/LLM-NeedleInAHaystack

https://x.com/LotusDecoder/status/2009809387951468787

提示这些模型在长上下文下可能存在能力边界，但具体在 agent 的许多场景中表现如何，还需要针对性测试。

有效上下文窗口 Gemini-3-pro 在40K，glm-4.7 在 35K。 根据github评测项目作者在大海捞针测试，Gemini-3-pro 在41k时断崖式崩塌到 20%多。意味着超出边界后，问牛答马，搞不清对象，这在复杂任务，高信息密度情况下，几乎是灾难。 然后我本地安排claude code测试了一次glm-4.7，发现ta在40K开始崩得更厉害，直接无法回复连贯有逻辑的文本。 现在 claude code 开机经常 20K，opencode 开机 23K，也就是说留给 难兄难弟们 ，保守只有15K用，这就决定了，他们跑不了多少轮会话和长任务，高信息密度任务。


选择 mcp or skill?

  • 目前来看一个偏执行一个偏文档
  • 未来mcp的设计很有可能引入skill，比如说出现专有化mcp server，该server上同时存在专用化script和reference（相当于把数据 + 获取数据的工具放在同类server上，资产化了说是）
