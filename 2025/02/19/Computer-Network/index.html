TCP

三次握手&四次挥手

三次握手

首要原因：防止「历史连接」初始化了连接

考虑场景：服务器与旧的SYN请求建立连接

次要原因：同步双方初始序列号

两次握手只保证了一方的初始序列号能被对方成功接收，没办法保证双方的初始序列号都能被确认接收。

次要原因：资源浪费

两次握手中，服务端不知道客户端是否收到了自己发送的ACK报文，必须假设其收到了并建立连接

四次挥手：双方各一个 FIN + ACK

客户端发送FIN包表示不再发送数据，但仍可接收

服务端发送ACK包表示收到

服务端发送FIN包表示自己侧数据已完成处理，开始关闭

客户端发送ACK表示收到，自己延时关闭；

为什么不是三次？

因为服务端在收到客户端想要关闭的请求时，自己可能仍在处理数据，没办法立刻关闭

如果服务端可以立即关闭，则可以当场回复FIN+ACK，实现三次挥手关闭连接

HTTP

HTTP1.1

长连接

建立TCP连接后保持

管道网络传输

在同一个TCP连接下，连续发送多个HTTP请求；要求对每一个请求进行回应；

如果服务端在处理 A 请求时耗时比较长，那么后续的请求的处理都会被阻塞住，这称为「队头堵塞」。

HTTP/1.1 管道解决了请求的队头阻塞，但是没有解决响应的队头阻塞；请求的队头阻塞是指，请求后必须等待收到响应才能发送新请求；

实际上，HTTP 管道化技术不是默认开启，后续对HTTP 1.1 的讨论基于管道化技术没有开启

Host字段

在HTTP1.0中认为每台服务器都绑定一个唯一的IP地址，因此，请求消息中的URL并没有传递主机名，但随着虚拟主机技术的发展，在一台物理服务器上可以存在多个虚拟主机，并且它们共享一个IP地址。

HTTP1.1的请求消息和响应消息都应支持Host头域，且请求消息中如果没有Host头域会报告一个错误（400 Bad Request）。

此外，服务器应该接受以绝对路径标记的资源请求。

Chunked Transfer Coding

HTTP/1.1将发送方将消息分割成若干个任意大小的数据块，每个数据块在发送时都会附上块的长度，最后用一个零长度的块作为消息结束的标志。

这种方法允许发送方只缓冲消息的一个片段，避免缓冲整个消息带来的过载。

Cache

HTTP/1.1在1.0的基础上加入了一些Cache的新特性，当缓存对象的Age超过Expire时变为Stable对象，Cache不需要直接抛弃Stable对象，而是与源服务器进行重新激活。

HTTP2.0

HTTP2.0和HTTP1.X相比的新特性

  • 二进制帧，HTTP1.x的解析是基于文本格式
  • 并发传输，一个 TCP 连接包含 多个 stream，每个 stream 包含多个 Message，每个 Message 包含多个 Frame
  • 头部压缩，HPACK 算法压缩头部
  • 服务端主动推送

数据流

HTTP/2 的数据包不是按顺序发送的，同一个连接里面连续的数据包，可能属于不同的回应。

因此，必须要对数据包做标记，指出它属于哪个回应。

每个请求或回应的所有数据包，称为一个数据流（Stream）。

每个数据流都标记着一个独一无二的编号，其中规定客户端发出的数据流编号为奇数， 服务器发出的数据流编号为偶数

客户端还可以指定数据流的优先级。优先级高的请求，服务器就先响应该请求。

同一个stream上的帧可以保证顺序性，将http报文拆成帧后考虑在同一个stream上传输？

HTTP2.0的多路复用和HTTP1.X中的长连接复用有什么区别

  • HTTP/1.1的Pipeling为若干个请求排队串行化单线程处理，后面的请求等待前面请求的返回才能获得执行机会，一旦有某请求超时等，后续请求只能被阻塞，毫无办法；
  • HTTP2.0多个请求可同时在一个连接上并行执行，某个请求任务耗时严重，不会影响到其它连接的正常执行

存在问题：tcp链接产生的阻塞

HTTP3.0

使用UDP协议

HTTP/2 主要的问题在于：多个 HTTP 请求在复用一个 TCP 连接，下层的 TCP 协议是不知道有多少个 HTTP 请求的。

所以一旦发生了丢包现象，就会触发 TCP 的重传机制，这样在一个 TCP 连接中的所有的 HTTP 请求都必须等待这个丢了的包被重传回来。

  • HTTP/1.1 中的管道传输中如果有一个请求阻塞了，那么队列后请求也统统被阻塞住了
  • HTTP/2 多请求复用一个TCP连接，一旦发生丢包，就会阻塞住所有的 HTTP 请求。

这都是基于 TCP 传输层的问题，所以 HTTP/3 把 HTTP 下层的 TCP 协议改成了 UDP！

QUIC协议-基于UDP的可靠传输协议

  • 无队头阻塞；

  • 更快的连接建立；

  • 连接迁移；

HTTPS

服务端返回客户端的认证内容为

服务器公钥 + 签名（CA使用CA的私钥对服务器公钥进行的计算的结果）

客户端使用预先保存的CA的公钥对签名进行解密，对比解密结果与公钥是否一致

HTTPS流程

TCP(3) + TLS(4)

  • Client发起一个HTTPS的请求
  • Server把事先配置好的公钥证书返回给客户端。
  • Client验证公钥证书：比如是否在有效期内，证书的用途是不是匹配Client请求的站点，是不是在CRL吊销列表里面，它的上一级证书是否有效，这是一个递归的过程，直到验证到根证书（操作系统内置的Root证书或者Client内置的Root证书），如果验证通过则继续，不通过则显示警告信息。
  • Client使用伪随机数生成器生成加密所使用的对称密钥，然后用证书的公钥加密这个对称密钥，发给Server。
  • Server使用自己的私钥解密这个消息，得到对称密钥。至此，Client和Server双方都持有了相同的对称密钥。
  • Server使用对称密钥加密明文内容A，发送给Client。
  • Client使用对称密钥解密响应的密文，得到明文内容A。
  • Client再次发起HTTPS的请求，使用对称密钥加密请求的明文内容B，然后Server使用对称密钥解密密文，得到明文内容B。

DNS

流程

  1. 本地主机查询 (Local Host File Check): 首先会检查本地的 hosts 文件 (位于不同操作系统上的不同位置，例如 Windows 的 C:\Windows\System32\drivers\etc\hosts 或 Linux/macOS 的 /etc/hosts) 是否包含 www.example.com 的 IP 地址。 如果找到，则跳过后面的步骤，直接使用该 IP 地址。 这是一个优先于 DNS 查询的优化机制。

  2. 递归查询本地 DNS 服务器 (Recursive Query to Local DNS Resolver): 如果 hosts 文件中没有找到，会向配置好的 本地 DNS 服务器 (Local DNS Resolver) 发送一个 递归查询 (Recursive Query)。 本地 DNS 服务器通常由 ISP (互联网服务提供商) 提供，也可能是手动配置的 (例如，Google Public DNS 或 Cloudflare DNS)。 “递归”意味着本地 DNS 服务器将负责完成整个解析过程，并最终返回 IP 地址。

  3. 递归查询根 DNS 服务器 (Recursive Query to Root DNS Servers): 如果本地 DNS 服务器不知道 www.example.com 的 IP 地址，它会联系其中一个 根 DNS 服务器 (Root DNS Servers)。 根 DNS 服务器知道所有顶级域名 (TLD) 服务器的地址。 全球共有 13 组根 DNS 服务器，它们具有不同的物理位置，并使用任播 (Anycast) 技术以提高可用性和效率。 本地 DNS 服务器会询问根服务器 www.example.com 的 IP 地址。 根服务器不会直接返回 IP 地址，而是返回负责 .com 域的 TLD DNS 服务器的地址。

  4. 递归查询 TLD DNS 服务器 (Recursive Query to TLD DNS Servers): 本地 DNS 服务器收到 .com TLD 服务器的地址后，会联系该 TLD 服务器，询问 www.example.com 的 IP 地址。 .com TLD 服务器也不会直接返回 IP 地址，而是返回负责 example.com 域的 权威 DNS 服务器 (Authoritative DNS Servers) 的地址。

  5. 递归查询权威 DNS 服务器 (Recursive Query to Authoritative DNS Servers): 本地 DNS 服务器收到 example.com 域的权威 DNS 服务器地址后，会联系这些权威 DNS 服务器，询问 www.example.com 的 IP 地址。 权威 DNS 服务器拥有 example.com 域的 DNS 记录，并知道 www.example.com 的 IP 地址。 它会将这个 IP 地址返回给本地 DNS 服务器。

    例如，假设 example.com 的权威 DNS 服务器有以下 DNS 记录：

    • example.com -> 192.0.2.1
    • bbb.example.com -> 192.0.2.2
    • aaa.example.com -> 192.0.2.3

    当本地 DNS 服务器查询 bbb.example.com 时，权威 DNS 服务器会返回 192.0.2.2。同样，当查询 aaa.example.com 时，权威 DNS 服务器会返回 192.0.2.3。

  6. 缓存与返回 (Caching and Return): 本地 DNS 服务器将收到的 IP 地址缓存起来，以便下次有人请求 www.example.com 时，它可以直接返回该 IP 地址，而无需再次执行上述步骤。 缓存时间由 DNS 记录的 TTL (Time To Live) 值决定。 最后，本地 DNS 服务器将 IP 地址返回给你的电脑。

  7. 连接服务器 (Connect to Server): 你的电脑现在有了 www.example.com 的 IP 地址，可以使用该 IP 地址连接到网站服务器，并获取网页内容。

JWT

JWT（JSON Web Token） 是一种开放标准（RFC 7519），用于在网络应用间安全地传递信息。JWT 是一个自包含的 Token，包含用户信息、签名和可选的其他元数据。

JWT 由三部分组成，用 . 分隔：

  • Header：包含 Token 的类型（JWT）和签名算法（如 HMAC SHA256 或 RSA）。

  • Payload：包含用户信息（如用户 ID、角色）和其他数据（如过期时间）。

  • Signature：对 Header 和 Payload 的签名，用于验证 Token 的完整性和真实性。

最终的 JWT 是一个 Base64Url 编码的字符串，格式如下：

Header.Payload.Signature

注意，JWT 由三部分组成：Header、Payload 和 Signature。其中：

  • Header 和 Payload 是 Base64Url 编码的明文，可以被任何人解码查看。
  • Signature 是对 Header 和 Payload 的签名，用于验证数据的完整性和真实性。

由于签名所使用的密钥仅服务器持有，因此可以通过该密钥确定签名的真实性

select, poll & epoll

select

位图+轮询 监控 文件描述符 集合

select 是一种经典的 I/O 多路复用技术，它允许程序监视多个文件描述符（FD），并在这些文件描述符上发生 I/O 操作时得到通知。它的工作原理和实现可以总结如下：

  • 工作方式：select 使用一个固定大小的位图（bitmap），标记哪些文件描述符需要检查。程序通过 select() 系统调用来监视这些文件描述符的状态变化（如是否可读、是否可写）。
  • 阻塞/非阻塞：select 可以设置为阻塞模式（一直等待直到有数据可用）或非阻塞模式（即不阻塞，立即返回）。
  • 文件描述符上限：select 在每次调用时，都需要传入一个文件描述符集合。由于它使用一个位图来表示文件描述符，因此其最大的文件描述符数量受到操作系统的限制，通常为 1024（可以通过修改系统配置增大，但通常仍有限制）。
  • 适用于小规模的文件描述符管理，容易实现，但由于它使用位图，文件描述符数目过多时会导致性能下降。
  • 由于每次都需要遍历所有的文件描述符，它的性能在大量文件描述符时可能非常低效。

poll

链表+轮询 监控 文件描述符 集合

poll 是对 select 的改进，解决了 select 存在的一些缺点，特别是在文件描述符上限和灵活性方面。

  • 工作方式：poll 也用来监视多个文件描述符，但是它不像 select 使用位图，而是使用链表（每个文件描述符一个 pollfd 结构体），其中每个元素包含文件描述符和该描述符的状态信息（如是否可读、是否可写等）。
  • 没有上限限制：与 select 的 1024 文件描述符的限制不同，poll 的文件描述符数目不受固定上限的限制，实际上只受系统内存限制。
  • 性能问题：虽然 poll 提高了灵活性，解决了文件描述符上限的问题，但它仍然需要遍历所有文件描述符，性能在大规模应用时依然较差。
  • 相对于 select，poll 更加灵活，没有文件描述符的数量限制，但仍然是线性扫描的方式，性能在大量文件描述符时依然不佳。

epoll

链表+事件回调 监控 文件描述符 集合

epoll 是 Linux 下特有的一种高效的 I/O 多路复用机制，它比 select 和 poll 在处理大量文件描述符时具有更高的效率。

  • 工作方式：epoll 提供了一种事件驱动的机制，注册感兴趣的文件描述符，当文件描述符发生事件时，epoll 通过回调函数通知应用程序。这使得它避免了每次都需要遍历所有文件描述符的低效问题。
  • 事件驱动：epoll 使用事件驱动模型，只关心发生事件的文件描述符。当事件发生时，内核会将文件描述符加入就绪队列，应用程序可以通过 epoll_wait 函数获取所有就绪的文件描述符，避免了像 select 和 poll 一样每次都遍历所有文件描述符。
  • 性能：epoll 在处理大量文件描述符时具有显著的性能优势，因为它采用了基于内存的事件回调机制，可以更高效地通知应用程序哪些文件描述符准备好进行 I/O 操作。
  • 支持边缘触发（Edge Triggered）和水平触发（Level Triggered）：epoll 提供了两种触发模式，边缘触发和水平触发，提供更多的灵活性和控制。
  • epoll 是在处理大量文件描述符时，性能最优的 I/O 多路复用技术。它基于事件驱动，避免了每次遍历所有文件描述符的低效操作，并提供了高效的通知机制。
