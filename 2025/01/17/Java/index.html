如果一个方法调用了另一个可能抛出受检异常（Checked Exception）的方法（即被调用的方法签名中使用了 throws 声明），那么调用方方法必须做以下两件事之一：

  1. 使用 try-catch 捕获并处理异常，或者
  2. 在自己的方法签名中使用 throws 声明该异常，将异常继续向上抛出。

Try-Catch-Finally

执行逻辑

Try块顺序执行，直到遇到Exception

即 1.若没有抛出Exception，则Try块将顺序执行完成

​ 2.若抛出Exception，Try块执行将到抛出Exception的位置截至

Catch块仅在Try块抛出Exception时执行

Finally块 不论 Try块/Catch块 的执行情况，一定会得到执行

即 即使在Try块/Catch块中存在return，Finally块一定会得到执行，且在Finally块执行完成后进行return；

泛型擦除：编译器在对源代码进行编译的时候将泛型换成了泛型指定的上限类<T extends 上限类型>，如果没有指定泛型的上限，编译器则会使用Object类替代。简单的说在编译完成后的字节码文件中其实是没有泛型的概念的，源代码中的泛型被编译器用Object或者泛型指定的类给替换掉了。

将所有泛型类型参数替换为其上限类型 (Upper Bound)。

  • 例如，List<String> 会被替换为 List<Object>，因为 Object 是 String 的上限类型。
  • 如果没有指定上限类型，则默认使用 Object 作为上限类型。

插入类型转换代码，以保证类型安全。

  • 例如，从 List<Object> 中获取元素时，编译器会插入 (String) 强制类型转换代码，以确保获取到的元素是 String 类型。

作用：声明了一个泛型为String的List集合，并向集合中添加了一个字符串“abc”，接着通过反射向集合list中添加了一个整数类型123，通过输出结果可以看到两个值都被添加到了集合中

类加载

概念

  1. 加载 (Loading)：

    • 通过类的全限定名（例如 java.lang.String）查找并读取类的二进制数据（.class 文件）。
    • 将类的静态结构信息（例如类名、方法名、字段名、访问权限等）存储在方法区中，并创建一个代表该类的 Class 对象，该Class对象内保存了指向方法区中这些结构信息的引用，该对象唯一。
    • 加载阶段通常由类加载器完成，Java 提供三种内置的类加载器：
      • Bootstrap ClassLoader：负责加载 Java 核心类库（rt.jar 等）。
      • Extension ClassLoader：负责加载 Java 扩展类库（jre/lib/ext 目录下的类）。
      • Application ClassLoader：负责加载应用程序类路径（CLASSPATH）上的类。

  2. 链接 (Linking)：

    • 验证 (Verification)：确保加载的类文件符合 Java 虚拟机规范，防止恶意代码破坏 JVM。

    • 准备 (Preparation)：为类的静态变量分配内存空间，并设置默认初始值（例如，int 类型为 0）。

    • 解析 (Resolution)：将符号引用（例如类名、方法名）转换为直接引用（例如内存地址）。

  3. 初始化 (Initialization)（非必定发生）：

    • 执行类的静态初始化代码块（static {}）和静态变量的初始化语句。
    • 初始化阶段类变量才会被赋予我们在代码中声明的值。JVM会根据语句执行顺序对类对象进行初始化。
    • 只有在类被主动使用时才会进行初始化，例如：
      • 创建类的实例。
      • 访问类的静态变量。
      • 调用类的静态方法。
      • 反射调用 Test 类。

初始化发生条件：

① 遇到 new、getstatic、putstatic、invokestatic 这四条字节码指令时，如果类没有进行过初始化，则需要先触发其初始化。生成这4条指令的最常见的Java代码场景是：使用new关键字实例化对象的时候、读取或设置一个类的静态字段（被final修饰、已在编译器把结果放入常量池的静态字段除外）的时候，以及调用一个类的静态方法的时候。

② 使用 java.lang.reflect 包的方法对类进行反射调用的时候，如果类没有进行过初始化，则需要先触发其初始化。

③ 当初始化一个类的时候，如果发现其父类还没有进行过初始化，则需要先触发其父类的初始化。

④ 当虚拟机启动时，用户需要指定一个要执行的主类（包含main()方法的那个类），虚拟机会先初始化这个主类。

⑤ 当使用 JDK1.7 动态语言支持时，如果一个 java.lang.invoke.MethodHandle实例最后的解析结果 REF_getstatic,REF_putstatic,REF_invokeStatic 的方法句柄，并且这个方法句柄所对应的类没有进行初始化，则需要先出触发其初始化。

⑥ 当一个接口中定义了JDK8新加入的默认方法（被default关键字修饰的接口方法）时，如果有这个接口的实现类发生了初始化，那该接口要在其之前被初始化。

JVM类加载机制

类加载器

  • 启动类加载器: Bootstrap ClassLoader，负责加载存放在JDK\jre\lib(JDK代表JDK的安装目录，下同)下，或被-Xbootclasspath参数指定的路径中的，并且能被虚拟机识别的类库(如rt.jar，所有的java.*开头的类均被Bootstrap ClassLoader加载)。启动类加载器是无法被Java程序直接引用的。
  • 扩展类加载器: Extension ClassLoader，该加载器由sun.misc.Launcher$ExtClassLoader实现，它负责加载JDK\jre\lib\ext目录中，或者由java.ext.dirs系统变量指定的路径中的所有类库(如javax.*开头的类)，开发者可以直接使用扩展类加载器。
  • 应用程序类加载器: Application ClassLoader，该类加载器由sun.misc.Launcher$AppClassLoader来实现，它负责加载用户类路径(ClassPath)所指定的类，开发者可以直接使用该类加载器，如果应用程序中没有自定义过自己的类加载器，一般情况下这个就是程序中默认的类加载器。
  • 自定义类加载器: 因为JVM自带的ClassLoader只是懂得从本地文件系统加载标准的java class文件，因此如果编写了自己的ClassLoader，便可以做到如下几点:
    • 在执行非置信代码之前，自动验证数字签名。
    • 动态地创建符合用户特定需要的定制化构建类。
    • 从特定的场所取得java class，例如数据库中和网络中。

类加载类型

  1. 全盘负责，当一个类加载器负责加载某个Class时，该Class所依赖的和引用的其他Class也将由该类加载器负责载入，除非显示使用另外一个类加载器来载入
  2. 父类委托，先让父类加载器试图加载该类，只有在父类加载器无法加载该类时才尝试从自己的类路径中加载该类
  3. 缓存机制，缓存机制将会保证所有加载过的Class都会被缓存，当程序中需要使用某个Class时，类加载器先从缓存区寻找该Class，只有缓存区不存在，系统才会读取该类对应的二进制数据，并将其转换成Class对象，存入缓存区。这就是为什么修改了Class后，必须重启JVM，程序的修改才会生效
  4. 双亲委派机制, 如果一个类加载器收到了类加载的请求，它首先不会自己去尝试加载这个类，而是把请求委托给父加载器去完成，依次向上，因此，所有的类加载请求最终都应该被传递到顶层的启动类加载器中，只有当父加载器在它的搜索范围中没有找到所需的类时，即无法完成该加载，子加载器才会尝试自己去加载该类。

双亲委派机制过程

  1. 当AppClassLoader加载一个class时，它首先不会自己去尝试加载这个类，而是把类加载请求委派给父类加载器ExtClassLoader去完成。
  2. 当ExtClassLoader加载一个class时，它首先也不会自己去尝试加载这个类，而是把类加载请求委派给BootStrapClassLoader去完成。
  3. 如果BootStrapClassLoader加载失败(例如在$JAVA_HOME/jre/lib里未查找到该class)，会使用ExtClassLoader来尝试加载；
  4. 若ExtClassLoader也加载失败，则会使用AppClassLoader来加载，如果AppClassLoader也加载失败，则会报出异常ClassNotFoundException。

  • 如果 Test.java 中的其他类没有被主类直接或间接使用，那么这些类 不会被加载 。

触发类加载的时候一般为：

  1. 创建 Test 类的实例：

    当执行到 new Test() 时，JVM 发现 Test 类还没有被加载，就会启动类加载过程。

    这种情况会一口气完成加载、链接和初始化三个过程。

    因为创建实例需要用到类的所有信息，包括静态变量的初始值、实例变量的定义等，所以必须完成初始化。

  2. 访问 Test 类的静态成员（变量或方法）：

    访问 Test.value 或调用 Test.someStaticMethod() 时，需要用到 Test 类的信息，会触发类加载。

    如果访问的是静态常量（final static），则只会进行加载和链接，不会进行初始化。

    • 因为静态常量在编译阶段就已经确定了值，并存储在类的常量池中，不需要等到初始化阶段。

    如果访问的是静态变量，则会完成加载、链接和初始化三个过程。

    • 因为静态变量的初始值可能需要在初始化阶段执行代码才能确定。

  3. 使用反射机制操作 Test 类：

    使用 Class.forName() 获取 Test 类的 Class 对象时，也会触发类加载。

    默认情况下，只会进行加载和链接，不会进行初始化。

    • Class.forName() 方法可以指定是否进行初始化，默认是不进行初始化的。

    如果调用 Class.forName() 方法时，第二个参数传入了 true，则会完成加载、链接和初始化三个过程。

    • 例如：Class.forName("com.example.Test", true, classLoader);

Java APT（注解处理器）是 Java 编译器提供的一种机制，它允许开发者在编译期处理注解，生成额外的代码或进行其他操作。

APT 的工作原理

  4. 编译生成最终的 class 文件: 编译器将生成的代码或资源文件与源代码一起编译，生成最终的 class 文件。

JMM

Java Memory Model，Java内存模型，关于多线程环境下的共享变量可见性、指令重排等方式的设计

Java使用共享内存并发模型

在共享内存并发模型中，多个线程或进程共享同一块内存区域，所有线程都可以读取和写入这些共享内存中的数据。为了确保多个线程或进程之间的同步，通常需要使用锁（如互斥锁、读写锁等）或其他同步机制。

存在其他并发内存模型：消息传递并发模型

在消息传递并发模型中，线程或进程通过发送和接收消息来进行通信。每个线程或进程都有自己的私有内存区域，不能直接访问其他线程或进程的内存。当需要交换信息时，它们通过消息传递机制来进行。这种模型不涉及共享内存，因此不需要锁等同步机制，通常更易于避免竞态条件。

特性

  • 可见性：当一个线程修改了共享变量的值，其他线程立即可见（通过 volatile 关键字实现）
  • 原子性：操作为一个整体，不会被其他操作打断（禁止对 volatile 变量的操作进行指令重排序）
  • 有序性：程序执行顺序按照代码先后顺序执行（通过关键字 synchronized 和 volatile 实现）

volatile 禁止指令重排

volatile 修饰变量要么在堆，要么在元空间

准确来说不是禁止了指令重排，而是确保volatile关键字修饰的变量的写入指令之前的指令重排后仍在其之前，volatile关键字修饰的变量的读取指令之后的指令重排后仍在其之后

  • 当程序执行到 volatile 变量的读操作或者写操作时，在其前面操作的更改肯定已经全部进行，且结果对后面的操作可见；在其后面的操作肯定还没有进行；
  • 在进行指令优化时，不能将 volatile 变量的语句放在其后面执行，也不能把 volatile 变量后面的语句放到其前面执行。

内存

  • 主内存：堆中对象实例
  • 工作内存：线程栈

Happens-Before 关系：

  • Happens-Before 关系是 JMM 中定义的一个重要概念，确保在某些操作发生之后，其他操作的结果是可见的。
  • 比如，某个线程对共享变量的写操作，在另一个线程的读操作之前发生，Java 保证第一个线程的写操作会在第二个线程的读操作之前“可见”。
  • JMM 定义了多个规则来确定操作之间的 Happens-Before 关系，例如：
    • 程序顺序规则：一个线程中的操作按代码顺序执行。
    • 锁规则：释放锁的操作 happens-before 后续的获取锁操作。
    • volatile 规则：对一个 volatile 变量的写操作 happens-before 后续对该变量的读操作。

ThreadLocal

每个线程类持有 ThreadLocalMap 实例

ThreadLocalMap 为 key-value结构

  • key 为 ThreadLocal 对象，弱引用

  • value 为 通过 threadLocal.set() 设置的值，强引用

由于线程池中的线程复用，线程任务结束后，可能出现 key 的 ThreadLocal 对象被回收，ThreadLocalMap 中的 value 仍然存在的情况。

由于 key 已经被回收，ThreadLocalMap 中的 entry 会变成 key = null, value = Object 的状态。这种情况下，value 无法被垃圾回收，因为线程仍然持有对 ThreadLocalMap 的引用，而 ThreadLocalMap 仍然引用着 Object 对象。如果 Object 对象占用较大内存（如 byte[] info 分配了 1MB），在长时间运行的应用中，这种内存泄漏会导致内存不断增长，最终引发 OutOfMemoryError。

锁

锁分类

公平锁与非公平锁

公平锁是指多个线程按照申请锁的顺序来获取锁，线程直接进入同步队列中排队，队列中最先到的线程先获得锁。非公平锁是多个线程加锁时每个线程都会先去尝试获取锁，如果刚好获取到锁，那么线程无需等待，直接执行，如果获取不到锁才会被加入同步队列的队尾等待执行。

排他锁与共享锁

排他锁也叫独占锁，是指该锁一次只能被一个线程所持有。如果线程T对数据A加上排它锁后，则其他线程不能再对A加任何类型的锁。获得排它锁的线程即能读数据又能修改数据。共享锁是指该锁可被多个线程所持有。如果线程T对数据A加上共享锁后，则其他线程只能对A再加共享锁，不能加排它锁。获得共享锁的线程只能读数据，不能修改数据。

可重入锁与非可重入锁

可重入锁 (ReentrantLock):

  • 定义: 同一个线程可以多次获取同一把锁，而不会导致死锁。
  • 实现机制: 可重入锁通常使用一个计数器来跟踪线程获取锁的次数。
    • 当线程第一次获取锁时，计数器加 1。
    • 如果同一个线程再次获取该锁，计数器继续加 1。
    • 当线程释放锁时，计数器减 1。只有当计数器变为 0 时，锁才会真正被释放，其他线程才能获取。
  • Java 中的实现: ReentrantLock 类、synchronized 关键字（隐式实现）。

非可重入锁 (Non-reentrant Lock):

  • 定义: 如果一个线程已经获取了某个锁，在释放该锁之前，不能再次获取该锁。
  • Java 中的实现: Java 中没有直接提供非可重入锁的实现，需要开发者自行实现。

乐观锁与悲观锁

乐观锁 (Optimistic Lock):

  • 定义: 乐观地认为并发冲突发生的概率较低，因此不会一开始就加锁，而是 在更新数据时检查是否发生了冲突。
  • 实现机制: 通常使用 版本号 或 时间戳 来检测冲突。
    • 在读取数据时，获取数据的版本号或时间戳。
    • 在更新数据时，比较当前版本号或时间戳与之前获取的是否一致。
    • 如果一致，则更新数据；如果不一致，则说明发生了冲突，需要进行处理（例如重试或回滚）。
  • Java 中的实现: 可以使用 AtomicInteger、AtomicLong 等原子类实现乐观锁，或者使用 CAS (Compare and Swap) 操作实现。

悲观锁 (Pessimistic Lock):

  • 定义: 悲观地认为并发冲突发生的概率较高，因此 在操作数据时都会先获取锁，防止其他线程同时访问或修改数据。
  • 实现机制: 通常使用 互斥锁 来实现，例如 Java 中的 synchronized 关键字和 ReentrantLock 类。
  • Java 中的实现: synchronized 关键字、ReentrantLock 类等。

可中断锁与不可中断锁

可中断锁：获取锁的过程中可以被中断，不需要一直等到获取锁之后 才能进行其他逻辑处理。ReentrantLock 就属于是可中断锁。

不可中断锁：一旦线程申请了锁，就只能等到拿到锁以后才能进行其他的逻辑处理。 synchronized 就属于是不可中断锁。

自用自出

第一次进入同步块时：

  • 当 Thread-1 第一次进入 exampleMethod() 方法时，会请求获取 exampleMethod() 的锁。此时，JVM 会为该线程创建一个偏向锁，并在 BiasedLockExample 实例的对象头的 Mark Word 中记录下 Thread-1 的线程 ID。

第二次进入同步块时：

  • 如果同一个线程 Thread-1 再次访问 exampleMethod() 方法，由于 Mark Word 中记录了它自己的线程 ID，JVM 会直接通过检查 Mark Word 来验证该线程已经持有锁。如果锁确实是偏向于 Thread-1，则无需使用 CAS 操作来加锁，直接进入方法。

其他线程尝试进入同步块时：

  • 当 Thread-2 尝试进入 exampleMethod() 方法时，它会看到 Mark Word

    中已经记录了 Thread-1 的线程 ID，这就表示有其他线程持有该锁。此时会进行如下的操作：

    • CAS 操作：JVM 会尝试使用 CAS 来将 Mark Word 中的线程 ID 替换成 Thread-2 的线程 ID。
    • 成功：如果 Thread-1 已经退出并释放了锁（例如它已经执行完毕），Mark Word 会成功更新为 Thread-2 的线程 ID。此时，锁仍然是偏向锁，Thread-2 可以继续执行该同步块的代码，不需要进行其他复杂的加锁操作。
    • 失败：如果 Thread-1 仍然持有锁（即它没有退出方法），CAS 操作会失败。此时， JVM 会在一个安全点暂停 Thread-1 的执行，撤销偏向锁与锁升级，恢复 Thread-1 的执行；此时，Thread-1 和 Thread-2 均不再持有该锁，该锁状态为轻量级锁无所属线程状态；线程 1 和线程 2 重新竞争轻量级锁， 接下来的锁竞争会使用轻量级锁的机制来进行（通过自旋等方式进行争抢）。

撤销偏向锁的过程：

  • JVM 会遍历 Thread-1 的栈帧，检查是否存在锁记录。如果存在，JVM 会清除与偏向锁相关的锁记录，并且会将 Mark Word 中保存的 Thread-1 的线程 ID 置为 0，表示没有线程持有该锁。
  • 重置偏向锁标识：偏向锁的标识位会被重置为 00，即表示没有持有锁，也没有任何线程占用该锁。

轻量级锁状态：

其他线程cas抢锁

JVM 会为每个线程在当前线程的栈帧中创建用于存储锁记录的空间，我们称为 Displaced Mark Word。如果一个线程获得锁的时候发现是轻量级锁，会把锁的 Mark Word 复制到自己的 Displaced Mark Word 里面。

然后线程尝试用 CAS 将锁的 Mark Word 替换为指向锁记录的指针。如果成功，当前线程获得锁，如果失败，表示 Mark Word 已经被替换成了其他线程的锁记录，说明在与其它线程竞争锁，当前线程就尝试使用自旋来获取锁。

轻量级锁的竞争过程：

  • 自旋：当一个线程（如 Thread-1 或 Thread-2）尝试获取轻量级锁时，它会首先检查 Mark Word，并通过 CAS 操作 尝试将 Mark Word 中的值修改为指向自己线程栈中的锁记录（通常是一个指向栈帧的指针）。
  • 如果 CAS 成功，该线程就成功获取锁，并且进入同步块。
  • 如果 CAS 失败（表示另一个线程已经成功获得了锁），线程会继续自旋，尝试再次获取锁。自旋操作是一种轻量级的争用机制，通常会在短时间内尝试多次，避免系统进入阻塞状态。

如果自旋失败：

  • 如果自旋的次数过多，或者多个线程持续争抢该锁，轻量级锁会升级为 重量级锁（修改锁标识位为10）；当线程2反复抢线程1持有的轻量级锁时，JVM会将该锁升级为重量级锁。升级过程中，JVM会强制暂停线程1，并撤销其持有的轻量级锁，然后两个线程会重新竞争这个重量级锁。

重量级锁状态：

当多个线程竞争同一个对象的锁，并且竞争激烈时，就会使用重量级锁（例如 ReentrantLock）。重量级锁会导致线程阻塞和上下文切换，性能开销较大。

线程与同步

LockSupport, wait()/notify() 和 join()

机制区别

  • LockSupport: 基于许可证机制，每个线程都有一个许可证，park() 消耗许可证，unpark() 颁发许可证。
  • wait()/notify(): 基于锁对象，线程需要先获取锁对象的监视器锁，才能调用 wait() 进入等待，notify() 唤醒等待相同锁对象的线程。
  • join(): 基于线程生命周期，让一个线程等待另一个线程执行完毕。

锁依赖

  • LockSupport: 不依赖于任何锁，可以直接阻塞和唤醒线程。
  • wait()/notify() 和 **join(): ** 都需要在获取锁的前提下才能使用，并且 wait()/notify() 还要求操作相同的锁对象。

精确性

  • LockSupport: 可以更精确地控制线程的阻塞和唤醒，例如可以指定要唤醒的线程。
  • wait()/notify(): notify() 方法随机唤醒一个线程，不够精确。
  • join(): 只能等待目标线程执行完毕，无法实现更细粒度的控制。

使用场景

  • LockSupport: 适用于更底层的线程同步场景，例如实现自定义同步工具类。
  • wait()/notify(): 适用于经典的生产者-消费者模型等场景。
  • join(): 适用于需要等待某个线程执行完毕后再继续执行的场景。

Thread.join()

谁调用ThreadX.join()，谁加入线程X的等待队列，等待线程X执行完成

对于线程 ThreadA 和线程 ThreadB:

若 ThreadA 在运行过程中执行 ThreadB.join()，

ThreadA 进入 WAITING 状态: ThreadB.join() 会让 ThreadA 立即进入 WAITING 状态，并且ThreadA 会无限期地等待 ThreadB 完成，直到 ThreadB 执行完毕才会继续执行。ThreadA 将释放任何已持有的锁。

ThreadB 状态可能不变: 和之前一样，ThreadB.join() 的调用不会直接改变 ThreadB 的状态。ThreadB 会按照其自身逻辑和状态继续执行或等待。

ThreadB 执行完毕: 当 ThreadB 执行完毕后，ThreadA 会被唤醒，从 WAITING 状态恢复到 RUNNABLE 状态，并尝试重新获取之前释放的锁（如果有），然后继续执行 ThreadB.join() 之后的代码。

Thread.join(1000)

若 ThreadA 在运行过程中执行 ThreadB.join(1000)，

ThreadA 进入 TIMED_WAITING 状态： join(1000) 会让 ThreadA 立即进入 TIMED_WAITING（超时等待）状态，最多等待 1000 毫秒（1 秒）。ThreadA 将释放任何已持有的锁。

ThreadB 状态可能不变： ThreadB.join(1000) 的调用不会直接改变 ThreadB 的状态。ThreadB 会按照其自身逻辑和状态继续执行或等待。

两种情况后的 ThreadA：

  • 情况一： ThreadB 在 1000 毫秒内执行完毕: ThreadA 会被唤醒，从 TIMED_WAITING 状态恢复到 RUNNABLE 状态，并尝试重新获取之前释放的锁（如果有），然后继续执行 ThreadB.join(1000) 之后的代码。
  • 情况二： ThreadB 在 1000 毫秒内没有执行完毕: 1000 毫秒后，ThreadA 会自动被唤醒，从 TIMED_WAITING 状态恢复到 RUNNABLE 状态，并尝试重新获取之前释放的锁（如果有），继续执行 ThreadB.join(1000) 之后的代码。

wait() 和 notify()

  • wait() 和 notify() 方法必须在同步代码块 (synchronized) 或同步方法中使用。

  • 调用 wait() 和 notify() 方法的对象必须是该同步代码块或同步方法所对应的锁对象。

Object.wait()

让当前线程（假设是 ThreadA）无限期地释放锁对象，并进入 WAITING 状态，直到其他线程（例如 ThreadB）调用 notify() 或 notifyAll() 方法唤醒它。

Object.wait(1000)

让当前线程（假设是 ThreadA）释放锁对象，并进入 TIMED_WAITING（超时等待）状态，最多等待 1000 毫秒。在这段时间内，有两种情况可以让 ThreadA 被唤醒：

  • 其他线程调用 notify() 或 notifyAll() 方法
  • 等待时间超过 1000 毫秒

Thread.sleep() 与 Object.wait() 对比

  • sleep() 方法没有释放锁，而 wait() 方法释放了锁 。
  • wait() 通常被用于线程间交互/通信，sleep()通常被用于暂停执行。
  • wait() 方法被调用后，线程不会自动苏醒，需要别的线程调用同一个对象上的 notify()或者 notifyAll() 方法。sleep()方法执行完成后，线程会自动苏醒，或者也可以使用 wait(long timeout) 超时后线程会自动苏醒。
  • sleep() 是 Thread 类的静态本地方法，wait() 则是 Object 类的本地方法。

Object.notify()

  • 作用: 随机唤醒一个正在等待相同锁对象的线程。
  • 注意：
    • notify() 方法不会立即释放锁。被唤醒的线程需要等待调用 notify() 的线程释放锁之后，才能真正地获取锁并继续执行。
    • 如果有多个线程正在等待同一个锁对象，notify() 方法只会随机唤醒其中一个，而其他线程仍然会继续等待。

Object.notifyAll()

  • 作用: 唤醒所有正在等待相同锁对象的线程。
  • 注意：
    • 和 notify() 方法一样，notifyAll() 方法也不会立即释放锁。
    • 被唤醒的线程需要竞争锁，只有成功获取锁的线程才能继续执行。

额外：使用synchronized和wait/notify实现线程的顺序执行

LockSupport

许可证机制

LockSupport 的许可证机制可以理解为每个线程都拥有一个虚拟的 “许可证”，每个线程最多只有一个许可证：

  • 初始状态： 线程刚创建时，默认没有许可证。
  • park() 行为：
    • 如果线程有许可证，调用 park() 会立即返回，并消耗掉这个许可证。
    • 如果线程没有许可证，调用 park() 会阻塞线程，直到获得许可证。
  • unpark() 行为：
    • 如果线程没有许可证，调用 unpark() 会颁发一个许可证给该线程。
    • 如果线程已经有许可证，调用 unpark() 不会产生任何效果，许可证不会累积。

LockSupport 的阻塞和唤醒

LockSupport 通过操作线程的 Parker 对象来实现阻塞和唤醒。每个线程都有一个 Parker 对象，它内部维护了线程的阻塞状态和许可证信息。

  • park() 阻塞线程： park() 方法会调用 Parker 对象的 park() 方法，将线程的状态设置为 WAITING 或 PARKED，并挂起线程，直到获得许可证。
  • unpark() 唤醒线程： unpark() 方法会调用 Parker 对象的 unpark() 方法，将线程的状态设置为 RUNNABLE，并通知操作系统调度该线程

状态与中断

BLOCKED 状态 (阻塞状态)

  • 成因: 线程在尝试获取某个锁时，如果该锁已经被其他线程持有，则当前线程会进入 BLOCKED 状态。
  • 中断的影响: BLOCKED 状态的线程不会响应中断请求。即使调用了 interrupt() 方法，线程也不会立即退出阻塞状态。只有当线程获取到锁之后，才会检查中断状态。

WAITING 状态 (等待状态)

  • 成因: 线程调用了 wait()、join()（无超时） 或 park() 方法，进入无限期等待状态。
  • 中断的影响: WAITING 状态的线程可以被中断唤醒。当调用 interrupt() 方法时，线程会抛出 InterruptedException，并退出等待状态。进入 RUNNABLE (可运行) 状态

TIMED_WAITING 状态 (超时等待状态)

  • 成因: 线程调用了 wait(timeout)、join(timeout)、sleep(timeout) 或 parkNanos(nanos) 方法，进入有限时间等待状态。
  • 中断的影响: TIMED_WAITING 状态的线程可以被中断唤醒。 当调用 interrupt() 方法时，线程会抛出 InterruptedException，并退出等待状态。进入 RUNNABLE (可运行) 状态

中断后状态

中断后程序会进入 RUNNABLE (可运行) 状态，但是不意味着程序可以立即向后执行代码

  • 需要获取资源: 线程需要重新获取之前释放的资源，例如：
    • 如果是通过 wait() 进入 WAITING 状态的，需要重新获取 wait() 方法所属对象的锁。此时线程若在 synchronized 块中，会等待获取对象的锁，若获取不到，会进入 BLOCKED 状态。
    • 如果是通过 join() 进入 WAITING 状态的，需要等待 join() 的目标线程执行完毕。
  • 需要 CPU 调度: 线程还需要等待 CPU 的调度，才能真正地执行代码。

Concurrent HashMap

ConcurrentHashMap 是 Java 提供的线程安全 Map 实现，其在 JDK 1.7 和 JDK 1.8 采用了不同的机制来提高并发性能：

JDK 1.7：分段锁（Segment Locking）

设计结构

在 JDK 1.7 版本中，ConcurrentHashMap 采用了 分段锁（Segment Locking） 机制，即 将整个 HashMap 拆分成多个小的 Segment（段），每个 Segment 维护一个独立的 HashMap。

每个 Segment 维护自己的 HashMap 和锁，所以不同的 Segment 之间可以并行访问，减少锁竞争。

主要操作

  1. put 操作
    • 计算 key 的 hash 值，定位到对应的 Segment。
    • 获取 Segment 的锁，执行 put 操作。
    • 释放锁。
  2. get 操作
    • 计算 key 的 hash 值，找到 Segment。
    • 不需要加锁，直接访问数据，提高性能。
  3. 扩容（rehash）
    • 只有一个 Segment 扩容，不影响其他 Segment，降低 rehash 的性能损耗。

JDK 1.8：CAS + Synchronized

从 JDK 1.8 开始，ConcurrentHashMap 移除了 Segment，改用 Node[] 作为底层数据结构，并结合 CAS（无锁操作）+ synchronized（锁竞争） 机制。

设计结构

  • 采用 数组 + 链表 + 红黑树 结构，类似 HashMap。
  • 数据分布在 Node[] 中，不再使用 Segment[]，简化结构。
  • 采用 CAS + synchronized 控制并发，提升性能。

主要优化

  1. put 操作
    • 如果目标桶（Node[]）为空，使用 CAS 操作 直接插入，避免加锁。
    • 如果目标桶有数据：
      • 若无并发冲突，采用 无锁操作 修改数据。
      • 若有并发冲突，使用 synchronized 锁定该桶，避免数据竞争。
    • 链表长度超过 8 时，自动转为红黑树，提高查询效率。
  2. get 操作
    • 直接读取 Node[]，无锁操作，提高性能。
  3. 扩容（rehash）
    • ConcurrentHashMap 采用 多线程并发扩容，不同线程可以同时帮助迁移数据，提升扩容效率。

在 Java 中，ConcurrentHashMap 和 HashMap 对于插入空键和空值的处理方式是不同的。

ConcurrentHashMap

  • 不允许插入空键：ConcurrentHashMap 不允许插入空键（null key）。如果你尝试插入一个空键，会抛出 NullPointerException。
  • 不允许插入空值：ConcurrentHashMap 也不允许插入空值（null value）。同样，如果你尝试插入一个空值，会抛出 NullPointerException。

原因：ConcurrentHashMap 的设计目标是支持高并发的场景。为了确保线程安全性和一致性，ConcurrentHashMap 在内部使用了复杂的锁机制和并发控制策略。允许空键或空值会增加实现的复杂性，并且可能导致歧义或不确定的行为。因此，ConcurrentHashMap 直接禁止了空键和空值的使用。

HashMap

  • 允许插入空键：HashMap 允许插入空键（null key）。你可以将一个键设置为 null，并且 HashMap 会将其视为一个有效的键。
  • 允许插入空值：HashMap 也允许插入空值（null value）。你可以将值设置为 null，并且 HashMap 会将其视为一个有效的值。

原因：HashMap 的设计目标是为单线程环境提供高效的键值对存储。它没有并发控制的需求，因此可以更灵活地处理空键和空值。允许空键和空值可以简化某些场景下的使用，例如在需要表示“不存在”或“未定义”的情况下。

FixedThreadPool的工作队列为无界队列LinkedBlockingQueue(队列容量为Integer.MAX_VALUE), 这会导致以下问题:

  • 线程池里的线程数量不超过corePoolSize,这导致了maximumPoolSize和keepAliveTime将会是个无用参数
  • 由于使用了无界队列, 所以FixedThreadPool永远不会拒绝, 即饱和策略失效

newSingleThreadExecutor

初始化的线程池中只有一个线程，如果该线程异常结束，会重新创建一个新的线程继续执行任务，唯一的线程可以保证所提交任务的顺序执行.

由于使用了无界队列, 所以SingleThreadPool永远不会拒绝, 即饱和策略失效

newCachedThreadPool

线程池的线程数可达到Integer.MAX_VALUE，即2147483647，内部使用SynchronousQueue作为阻塞队列； 和newFixedThreadPool创建的线程池不同，newCachedThreadPool在没有任务执行时，当线程的空闲时间超过keepAliveTime，会自动释放线程资源，当提交新任务时，如果没有空闲线程，则创建新线程执行任务，会导致一定的系统开销； 执行过程与前两种稍微不同:

  • 主线程调用SynchronousQueue的offer()方法放入task, 倘若此时线程池中有空闲的线程尝试读取 SynchronousQueue的task, 即调用了SynchronousQueue的poll(), 那么主线程将该task交给空闲线程. 否则执行(2)
  • 当线程池为空或者没有空闲的线程, 则创建新的线程执行任务.
  • 执行完任务的线程倘若在60s内仍空闲, 则会被终止. 因此长时间空闲的CachedThreadPool不会持有任何线程资源

拒绝策略

handler 线程池的饱和策略，当阻塞队列满了，且没有空闲的工作线程，如果继续提交任务，必须采取一种策略处理该任务，线程池提供了4种策略:

  • AbortPolicy: 直接抛出异常，默认策略；
  • CallerRunsPolicy: 用调用者所在的线程来执行任务；
  • DiscardOldestPolicy: 丢弃阻塞队列中靠最前的任务，并执行当前任务；
  • DiscardPolicy: 直接丢弃任务；

配置线程池需要考虑因素

从任务的优先级，任务的执行时间长短，任务的性质(CPU密集/ IO密集)，任务的依赖关系这四个角度来分析。并且近可能地使用有界的工作队列。

性质不同的任务可用使用不同规模的线程池分开处理:

  • CPU密集型: 尽可能少的线程，N_cpu+1
  • IO密集型: 尽可能多的线程, N_cpu*2，比如数据库连接池
  • 混合型: CPU密集型的任务与IO密集型任务的执行时间差别较小，拆分为两个线程池；否则没有必要拆分

线程池关闭

shutdown()

  • 功能: 启动一个平滑关闭过程。
  • 行为:
    • 阻止新任务提交到线程池。
    • 允许线程池中已提交的任务（包括正在执行和等待执行的任务）完成。
    • 线程池不会立即停止执行，而是等待所有已提交的任务执行完毕后，才真正关闭。
  • 返回值: 无返回值。
  • 使用场景: 适用于希望线程池优雅关闭，确保所有已提交的任务都得到处理的情况。

shutdownNow()

  • 功能: 尝试强制关闭线程池。
  • 行为:
    • 阻止新任务提交到线程池。
    • 尝试停止所有正在执行的任务（通过中断线程的方式）。
    • 跳过所有等待执行的任务，并返回一个包含这些未执行任务的 List。
  • 返回值: 返回 List<Runnable>，其中包含在 shutdown 时等待执行的任务列表。
  • 使用场景: 适用于需要立即停止线程池，而不关心未完成任务的情况。例如，在应用程序遇到严重错误需要立即退出时。

堆内存划分

Java 的堆内存主要被划分为两个区域：

  • 新生代 (Young Generation) 4 Minor GC: 存放新创建的对象。由于大多数对象的生命周期都比较短，因此新生代的空间相对较小，垃圾回收也比较频繁。新生代又可以细分为三个区域：
    • Eden 区: 新创建的对象首先被分配到这里, 空间不足时触发Minor GC
    • Survivor 区 (From, To) 4 Major GC: 经历过一次 Minor GC 后仍然存活的对象会被复制到 Survivor 区。新生代有两个 Survivor 区，每次 Minor GC 只使用其中一个，另一个处于闲置状态。复制算法
  • 老年代 (Old Generation): 存放生命周期较长的对象，例如缓存数据、数据库连接等。老年代的空间比较大，垃圾回收频率也相对较低。标记-清除/标记-整理算法

垃圾收集类型

传统垃圾收集器（大部分阶段）会触发STW，现代低延迟垃圾收集器（小阶段）触发STW

Java 中的垃圾收集主要分为两大类：

  • 部分收集 (Partial GC): 只收集部分堆内存空间，频率较高。

    • 新生代收集 (Minor GC/Young GC): 回收 Eden 区和一个 Survivor 区 (From Survivor 区) 的对象。

      触发条件：

      • 当 Eden 区空间不足时就会触发 Minor GC。

    • 老年代收集 (Major GC/Old GC): 收集老年代的垃圾。

      触发条件：

      • 老年代空间不足。
      • Minor GC 后存活的对象晋升到老年代，导致老年代空间不足。

  • 整堆收集 (Full GC): 收集整个堆内存空间的垃圾，包括新生代，老年代，永久代/元空间。Full GC 的频率很低，但是执行时间比较长，会造成应用程序的停顿。

JDK6 update 24 后回收规则

在 JDK6 update 24 之后，为了更好地控制 Full GC 的触发时机，引入了新的回收规则：

只要老年代的连续空间大于新生代空间总大小或者历次晋升的平均大小，就会进行 Minor GC，否则将进行 Full GC。

  • 老年代的连续空间: 指老年代中最大的一块连续可用空间。
  • 新生代空间总大小: 指 Eden 区和两个 Survivor 区的总大小。
  • 历次晋升的平均大小: 指历次 Minor GC 后晋升到老年代的对象的平均大小。

该规则的意义:

  • 尽可能避免 Full GC 的发生，因为 Full GC 的成本很高。
  • 当老年代的连续空间不足以容纳新生代的对象时，即使进行 Minor GC 也无法避免 Full GC，因此直接进行 Full GC 反而效率更高。

永久代垃圾回收

永久代内存区域也会进行垃圾回收。永久代的垃圾收集主要包废弃常量和无用的类（被类加载器卸载的Class）。永久代触发垃圾回收的条件比较困难，需要同时满足以下三点：

  • 该类所有的实例都已经被回收，也就是Java堆中不存在该类的任何实例；
  • 加载该类的ClassLoader已经被回收；
  • 该类对应的java.lang.Class 对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法；

对象晋升

当 Eden 区空间不足时，会触发 Minor GC。Minor GC 会将 Eden 区中存活的对象复制到 Survivor 区。如果 Survivor 区也满了，则会将部分存活的对象晋升到老年代。

晋升规则:

  • 每个对象都有一个年龄计数器，每经历一次 Minor GC，年龄计数器就加 1。当对象的年龄达到一定阈值 (默认是 15) 时，就会被晋升到老年代。
  • 如果 Survivor 区中相同年龄的对象总大小超过 Survivor 区空间的一半，则年龄大于等于该年龄的对象也会被晋升到老年代。！！！

垃圾回收算法

在垃圾回收过程中，应用软件将处于一种Stop the World的状态。在Stop the World状态下，应用程序所有的线程都会挂起，暂停一切正常的工作，等待垃圾回收的完成。如果垃圾回收时间过长，应用程序会被挂起很久，将严重影响用户体验或者系统的稳定性。

标记算法：引用计数算法

Java垃圾回收没有采用该方法

引用计数算法（Reference Counting）比较简单，对每个对象保存一个整型的引用计数器属性。用于记录对象被引用的情况。

对于一个对象A，只要有任何一个对象引用了A，则A的引用计数器就加1；当引用失效时，引用计数器就减1。只要对象A的引用计数器的值为0，即表示对象A不可能再被使用，可进行回收。

优点：实现简单，垃圾对象便于辨识；判定效率高，回收没有延迟性。

缺点：

  1. 它需要单独的字段存储计数器，这样的做法增加了存储空间的开销。
  2. 每次赋值都需要更新计数器，伴随着加法和减法操作，这增加了时间开销。
  3. 引用计数器有一个严重的问题，即无法处理循环引用的情况。这是一条致命缺陷，导致在Java的垃圾回收器中没有使用这类算法。

标记算法：可达性分析算法

Java垃圾回收使用了该算法

可达性分析算法是以根对象集合（GCRoots）为起始点，按照从上至下的方式搜索被根对象集合所连接的目标对象是否可达。

使用可达性分析算法后，内存中的存活对象都会被根对象集合直接或间接连接着，搜索所走过的路径称为引用链（Reference Chain）

如果目标对象没有任何引用链相连，则是不可达的，就意味着该对象己经死亡，可以标记为垃圾对象。

在可达性分析算法中，只有能够被根对象集合直接或者间接连接的对象才是存活对象。

清除算法：标记-清除算法

当堆中的有效内存空间（available memory）被耗尽的时候，就会停止整个程序（也被称为stop the world），然后进行两项工作，第一项则是标记，第二项则是清除

  1. 标记：Collector从引用根节点开始遍历，标记所有被引用的对象。一般是在对象的Header中记录为可达对象。
    • 注意：标记的是被引用的对象，也就是可达对象，并非标记的是即将被清除的垃圾对象
  2. 清除：Collector对堆内存从头到尾进行线性的遍历，如果发现某个对象在其Header中没有标记为可达对象，则将其回收

被清除的对象实际上是被加入到了空闲内存列表，下次分配内存时该区域为可用区域，直接覆盖写入新内容

清除算法：复制算法

将活着的内存空间分为两块，每次只使用其中一块，在垃圾回收时将正在使用的内存中的存活对象复制到未被使用的内存块中，之后清除正在使用的内存块中的所有对象，交换两个内存的角色，最后完成垃圾回收

新生代垃圾回收使用复制算法，Eden区和S0区存活对象整体复制到S1区

清除算法：标记-压缩算法

标记-压缩算法 等价于进行 标记-清除算法+内存碎片整理

  1. 第一阶段和标记清除算法一样，从根节点开始标记所有被引用对象
  2. 第二阶段将所有的存活对象压缩到内存的一端，按顺序排放。之后，清理边界外所有的空间。

常见垃圾收集器

年轻代垃圾收集器

Serial GC

  • 特点：单线程、使用“复制算法”进行垃圾回收。
  • 工作方式：仅使用一个线程进行垃圾收集，会暂停所有应用线程（Stop-The-World，STW）。
  • 优点：简单高效，适合单线程或小内存环境。
  • 适用场景：桌面应用、小型应用、单核CPU。
  • 配置参数：-XX:+UseSerialGC。

ParNew GC

  • 特点：多线程版的Serial GC，使用“复制算法”。
  • 工作方式：通过多线程并行回收年轻代，依然存在STW。
  • 优点：适合多核CPU环境，常与老年代的CMS GC搭配。
  • 适用场景：多线程应用程序。
  • 配置参数：-XX:+UseParNewGC。

Parallel Scavenge GC

  • 特点：多线程、注重吞吐量，使用“复制算法”。
  • 工作方式：多线程并行回收，优化吞吐量（GC时间与应用运行时间的比值）。
  • 优点：可调节GC暂停时间与吞吐量之间的平衡。
  • 适用场景：需要高吞吐量的后台应用程序。
  • 配置参数：-XX:+UseParallelGC。
  • 常用调优参数：
    • -XX:MaxGCPauseMillis=<N>：设置最大GC暂停时间。
    • -XX:GCTimeRatio=<N>：设置吞吐量比例。

老年代垃圾收集器

Serial Old GC

  • 特点：Serial GC的老年代版本，使用标记-整理算法。
  • 工作方式：单线程，存在STW。
  • 适用场景：单线程或小型应用，或与年轻代的Serial GC搭配。
  • 配置参数：-XX:+UseSerialOldGC。

CMS (Concurrent Mark-Sweep) GC

  • 特点：低延迟、并发回收，使用标记-清除算法。
  • 工作方式：通过多个阶段（初始标记、并发标记、重新标记、并发清除）回收垃圾。
  • 优点：降低STW时间，适合低延迟应用。
  • 缺点：可能产生碎片化，需要触发Full GC。
  • 适用场景：对响应时间敏感的应用。
  • 配置参数：-XX:+UseConcMarkSweepGC。
  • 常用调优参数：
    • -XX:CMSInitiatingOccupancyFraction=<N>：设置老年代使用率达到多少时触发GC。

整堆垃圾收集器

G1 (Garbage First) GC

  • 特点：分区收集，使用混合算法（标记-整理为主，标记-清除为辅）。
  • 工作方式：将堆划分为多个独立的区域（Region），并优先回收垃圾最多的区域。
  • 优点：可控的GC暂停时间，减少碎片化。
  • 适用场景：需要大内存且要求低延迟的应用。
  • 配置参数：-XX:+UseG1GC。
  • 常用调优参数：
    • -XX:MaxGCPauseMillis=<N>：设置目标最大暂停时间。

ZGC (Z Garbage Collector)

  • 特点：超低延迟、使用标记-移动算法，几乎不受堆大小影响。
  • 工作方式：通过并发标记和整理，确保STW时间通常在几毫秒内。
  • 优点：支持非常大的堆内存（TB级），适合极低延迟需求。
  • 缺点：内存占用稍高。
  • 适用场景：金融系统、在线交易系统等高实时性场景。
  • 配置参数：-XX:+UseZGC。
  • 常用调优参数：
    • -XX:MaxHeapSize=<size>：设置最大堆内存。

Shenandoah GC

  • 特点：低延迟、并发回收，使用标记-整理算法。
  • 工作方式：与ZGC类似，目标是最小化STW时间。
  • 优点：支持大堆内存，延迟极低。
  • 适用场景：高实时性应用。
  • 配置参数：-XX:+UseShenandoahGC。
